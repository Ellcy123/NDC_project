# State 可视化器叙事注解设计

适用对象：State 文件可视化工具（解析 YAML 格式的 loop_state 文件，面向人类策划）
文档版本：2026-04-08

---

## 设计原则

注解的目的是让策划**一眼看懂关键节点**，而不是给每个节点都贴标签。
原则：**少而精**。一个 Loop 里注解不超过 5 个，每个注解不超过 2 句话。

注解只出现在以下情况之一：
- 这个节点是玩家情感投入的转折点
- 这个节点的数据连接关系不直观，需要点睛
- 这个节点如果搞错了，后续 Loop 会出现连锁问题

## 第一部分：情绪弧线关键节点定义

### 1.1 节点类型与触发规则

#### 高潮节点（CLIMAX）

**定义：** 指证最后一轮击穿、NPC 完整坦白的节点。

**触发条件（从 state 数据判断）：**
- `expose.round_N` 是当前 Loop 的最后一轮（`id == total_rounds`）
- 且该轮的 `result` 字段中包含 NPC 供出核心真相的内容
- 且 `post_expose_knowledge` 列表非空（确认该 Loop 确实有揭示层）

**标注规则：** 每个 Loop 最多标一个高潮节点，永远在最后一轮指证上。

**示例（loop1 Unit1）：**
expose.round_3 → Rosa 最终供出 Morrison
注解：「本循环的情感顶点。Rosa 三轮被逼退后的崩口——她选的不是被迫，是恐惧耗尽。」

---

#### 转折节点（TURN）

**定义：** 玩家以为在往某个方向走，但在这里认知被翻转。

**触发条件（从 state 数据判断）：满足以下任意一条）**
- NPC 的 `lie.truth` 揭示的内容，与玩家 `known_facts` 中既有认知**方向相反**（不只是补充信息，而是否定旧认知）
- 某个 `expose.round_N.result` 中出现 NPC "退守新谎言"的记录（表明谎言结构是多层的，上一轮以为击穿了但没有）
- `player_context.new_directions` 中的方向，与当前 `known_facts` 的隐含假设矛盾（如 known_facts 隐含"只有一个人在楼顶"，但 new_direction 是"第二组脚印"）

**标注规则：** 每个 Loop 最多标两个转折节点。优先标信息方向的翻转，而非单纯的惊喜。

**示例（loop4 Unit3）：**
expose.round_1 → 脚印深浅不一真相揭晓（两个女人，不是一个）
注解：「Loop 1 埋的藏梗在这里引爆。策划需确认 L1 脚印描述没有提前破口。」

**示例（loop4 Unit3）：**
expose.round_2 → Helen"心软"说法出现
注解：「表面是坦白，实为退守。从'不在场'退到'在场但只是旁观'的路径，需要 R3 继续击穿。」

---

#### 信息交叉验证点（CROSS）

**定义：** 两条独立来源的信息在这里完成交叉，共同指向同一个结论。这是疑点解锁的必要条件。

**触发条件（从 state 数据判断）：**
- `doubts[].unlock_condition` 中同时包含 `testimony:` 和 `item:` 两种来源
- 或 `expose.round_N.usable_evidence` 列表中有 2 件以上证据，且这些证据来自**不同场景**（scene_id 不同）
- 或某轮指证中既使用了当前 Loop 新获取的证据，又口头引用了前序 Loop 的证据（state 注释中有 `L{N}已获得` 标记）

**标注规则：** 标在 `doubts` 解锁条件上，而不是标在具体证据条目上。每个 Loop 最多标三个交叉验证点。

**示例（loop1 Unit1）：**
doubt.1102 → 解锁条件 `item:1701 + item:1121`
注解：「毛巾（储物间）+ 氯仿瓶（走廊）来自不同房间，两件才能构成迷晕行为的完整证据链。」

**示例（loop4 Unit3）：**
expose.round_2 → `item:3408`（Helen 家 Jameson）+ 口头引用 `item:3107`（楼顶空瓶，L1 获得）
注解：「跨循环交叉：L1 的楼顶酒瓶到 L4 才找到来源。策划须确认 L1 物品描述未提前透露品牌来源。」

---

#### 伏笔埋点（PLANT）

**定义：** 当前 Loop 故意留下但不解释的细节，设计上预期在后续 Loop 揭晓。

**触发条件（从 state 数据判断）：**
- `doubts[].clear == false`（该疑点本 Loop 不解决，遗留至后续）
- 或 `expose.round_N.result` 中出现 `🔒` 标记（设计者标注的"开放性结论"）
- 或某 NPC 的 `does_not_know` 列表中出现与下一 Loop 核心主题相关的内容（需配合下一 Loop 的 `player_context.goals` 手动比对）
- 或 `known_facts` 中有 `（🔒藏梗：...Loop N揭晓）` 标记

**标注规则：** 伏笔埋点不标在"答案"上，标在"被故意留下的疑问"上。每个 Loop 最多标两个。标注时同时注明"预计在 Loop N 揭晓"。

**示例（loop1 Unit3）：**
doubt.关于脚印深浅不一
注解：「故意不解释的细节，Loop 4 揭晓真相（两个体重不同的女人）。策划须确认 L1-L3 中任何路径都不能提前给出答案。」

**示例（loop4 Unit3）：**
expose.round_3.result 中的 🔒 开放性结论（Helen 是否真的"心软"等）
注解：「4 个开放性问题是 L5 的引子，不是本循环需要回答的。」

---

#### 情感伏击点（AMBUSH）

**定义：** 玩家以为在追查真相，但在这里被角色的人性或处境击中——不是信息揭示，是情感冲击。

**触发条件（从 state 数据判断）：**
- NPC 的 `motive` 字段涉及家庭、恐惧、生存压力等非理性动机
- 且该 NPC 的 `expose.result` 中包含个人情感细节（孩子、遗物、童年等）
- 且该 NPC 的 `lie.truth` 与其 `motive` 之间存在悲剧性张力（为了保护某人而说谎、被迫参与等）

**标注规则：** 只标在指证的最终坦白处，且 motive 必须有具体情感内容（不能是纯粹的"利益驱动"）。每个 Loop 最多标一个。

**示例（loop1 Unit1）：**
expose.round_3 → Rosa 提到 Morrison 是警察、她见过他对付不听话的人、女儿病重
注解：「Rosa 说谎不是恶意，是活下去的本能。玩家此刻的情绪不应该是胜利感，而是复杂。」

**示例（loop4 Unit3）：**
expose.round_3 → Helen 提到 Emily（Mary 的女儿，8 岁）作为隐瞒的原因之一
注解：「Helen 用孩子解释沉默——比任何证据都更难反驳。策划须确认这里对话节奏给玩家留了停顿空间。」

---

### 1.2 节点标注汇总规则

| 节点类型 | 标注位置 | 每 Loop 上限 | 核心判断依据 |
|---------|---------|------------|------------|
| CLIMAX 高潮 | 最后一轮指证 result | 1 | 最后一轮 + post_expose_knowledge 非空 |
| TURN 转折 | 认知被翻转的证据/result | 2 | 否定旧认知，不只是补充信息 |
| CROSS 交叉验证 | doubts 解锁条件 | 3 | 两个以上不同来源的信息交汇 |
| PLANT 伏笔 | 未解决的疑点/🔒标记 | 2 | clear==false 或 🔒 明确标注 |
| AMBUSH 情感伏击 | 坦白中的人性细节 | 1 | motive 有具体情感内容且有悲剧张力 |

## 第二部分：大白话注解模板

### 注解写作原则

- 不解释机制，只点出"这里为什么重要"
- 语气平实，像一个有经验的策划在旁边看着另一个策划工作
- 避免"重要！""关键！"这类空洞强调词——用具体细节代替
- 第一句说事实，第二句说影响（对玩家体验或对后续设计的影响）

---

### 2.1 CLIMAX 高潮注解模板

**格式：**
「[NPC名] 在 [具体触发条件] 之后开口。[坦白的核心内容一句话]。」

**示例：**
「Rosa 在"Morrison 不知道你看到了"这句话之后开口。她保持沉默的原因消失了，但她的恐惧没有消失——这一点对话节奏要体现出来。」

「Helen 在 Margaret 目击证词击穿"谁都没碰他"之后完整坦白。从 L3 就开始守的"不在场"防线到这里才彻底崩塌，坦白的分量需要对话撑起来。」

---

### 2.2 TURN 转折注解模板

**格式：**
「玩家此前假设 [旧认知]，在这里发现 [翻转内容]。[需要注意的设计细节]。」

**示例：**
「玩家此前假设楼顶只有一个女人，在这里发现脚印深浅不一源于两个体重不同的人。L1-L3 的描述必须做到让这个翻转成立而不显得是 trick——脚印深浅在前三个 Loop 需要一直是开放的疑问，不能有哪句描述暗示"只有一人"。」

「玩家此前以为击穿了 Rosa 的谎言（她承认目击），但 R2 之后 Rosa 升级谎言声称"只是旁观者"。这个退守结构是设计意图，R3 需要再给一次击穿机会。」

---

### 2.3 CROSS 交叉验证注解模板

**格式：**
「[证据A]（来源：[场景/循环]）和 [证据B]（来源：[场景/循环]）在这里交汇，共同指向 [结论]。缺少任何一件都不足以定案。」

**示例：**
「工作记录卡（Rosa 储物间）和 Rosa 的证词（酒吧大堂）在这里交汇，共同指向"Rosa 说谎了自己的位置"。缺少任何一件都不足以定案——工作记录卡单独只说明安排，证词单独只说明主张。」

「门把手外侧油迹照片（L4 楼顶搜证）和地面油比对报告（L3 获得，系统层派生）在这里交汇，证明进入楼顶的人和涂油的人是同一个人。注意这是跨 Loop 交叉，L3 的物品描述不能提前暗示这个结论。」

---

### 2.4 PLANT 伏笔注解模板

**格式：**
「[具体细节] 在本 Loop 故意不解释。预计 Loop [N] 揭晓。在此之前，该细节在所有对话/描述中应保持"存疑但无答案"的状态。」

**示例：**
「脚印深浅不一在 L1 故意不解释，Mary 的 L2 辩词"地面不平整"是一个听起来勉强但能糊弄过去的解释，维持疑问状态到 L4。设计检查点：L1-L3 中确认没有任何描述暗示"两个人"的存在。」

「L4 最后的四个开放性问题（Helen 是否真的"心软"、Thomas 0.25% 是否还有攻击能力等）在本 Loop 故意不解答。这些是 L5 的线索引子，对话中可以让 Zack 有明确的困惑情绪，但不给结论。」

---

### 2.5 AMBUSH 情感伏击注解模板

**格式：**
「[NPC名] 在这里说出的不是证据，而是 [情感内容的一句话概括]。玩家对 [结果] 的情绪此时应该是 [预期情绪]，而不是 [需要避免的情绪]。」

**示例：**
「Rosa 在这里说出的不是证据，而是一个母亲的生存算计：Morrison 是警察，她见过他怎么对付不听话的人，女儿还在病床上。玩家在这里的情绪应该是复杂的理解，而不是审判者的胜利感——对话节奏要给这种复杂感留空间。」

「Helen 在这里说出的不是证据，而是一个童年目睹父亲打母亲的女人，面对另一个母亲和她的 8 岁女儿所做的选择。玩家此刻的情绪应该是"我能理解但我不确定这是否构成正当防卫"，而不是单纯的同情——这种纠结正是 L5 存在的理由。」

---

### 2.6 场景解锁状态注解（辅助型，非关键节点）

这类注解不标关键节点，而是在场景列表中用极简文字说明场景状态，帮助策划快速定位哪些场景有内容。

**状态标签（来自 state 文件已有的 emoji 惯例）：**
- 🎬 开篇场景（initScene）
- 🔓 可搜证/有新内容
- ⚔️ 指证场所
- ⏸️ 无新线索（物理上可进入但无新内容）
- 🔒 本 Loop 不可进入

**注解格式：** 只在 🔒 和 ⏸️ 场景上加一句解释原因，其他状态自明。

「SC3004 警局 🔒：L4 不可进入。Mickey 的介入发生在 closing event，不需要单独场景。」
「SC3009 Continental 银行 ⏸️：L4 无新证据。L3 证据继承，银行的角色在 L5 才正式展开。」

## 第三部分：指证拆解的展示节奏

### 3.1 整体结构

每次指证由三层构成：**谎言层 → 击穿层 → 退守层**（最后一轮是完整坦白，无退守）。

可视化器在展示一轮指证时，应呈现这三层的节奏，而不是把所有信息平铺。

```
轮次 N
├── 谎言（NPC 声称的）
│   └── 谎言来源（证词 ID / 上轮退守）
├── 可用证据
│   └── 每件证据：ID + 用途（一句话）
├── 击穿结果
│   ├── NPC 的反应（从 result 提取）
│   ├── 关键台词或行为（如果 result 有引号内容）
│   └── 退守谎言（如果有）→ 标注"进入下一轮"
└── 注解（仅关键节点有）
```

### 3.2 谎言 vs 真相的对比展示

**不要把"谎言"和"真相"放在同一层级展示**，这会提前破梗。

正确展示逻辑：
- 谎言：在每轮开头显示（这是玩家此时已知的 NPC 说法）
- 真相：只在最后一轮坦白的 `result` 中呈现，或在 `expose` 节点的整体概览中用折叠/隐藏方式处理

**对比展示的触发时机：** 当一轮指证的 `result` 中 NPC 明确承认某件事时，才在该轮展示"NPC 声称 X → 证据显示 Y → NPC 承认 Z"的三步流。

**示例展示文案（loop1 Unit1 R1）：**
```
第一轮：工作地点矛盾
Rosa 说：「从 9 点半开始在地下室打扫」
证据：工作记录卡（1115）——记录显示安排在后台走廊，不是地下室
结果：Rosa 无法解释，辩称"记录可能搞错了"→ 退守至下一轮
```

**示例展示文案（loop4 Unit3 R2）：**
```
第二轮：Thomas 喝不起的酒
Helen 说：「他自己在那喝酒，自己带的酒」（R1 退守）
证据 A：Helen 家 Jameson（3408）——同品牌，已开封剩 1/3
证据 B：Thomas 家廉价酒照片（3217）——全是 $0.10-0.15，欠 $12,000 买不起 $3 的 Jameson
结果：Helen 含泪承认带酒上楼顶，持续灌酒 → 退守至"Thomas 太醉了自己滑倒"
```

### 3.3 证据"击破"效果的可视化建议

每件证据有两种功能，展示时应有区分：

**直接击破（Direct）：** 证据直接与谎言矛盾，玩家选择这件证据即可进入下一状态。
- 视觉标记建议：实线连接"谎言 → 证据 → 结果"

**辅助引用（Support）：** 证据本身不足以击破，但在击破后被 Zack 口头引用来加深结论。
- 视觉标记建议：虚线或括号标注"口头引用"
- 来源标注：如果是前序 Loop 的证据，标注"L{N} 获得"

**跨 Loop 证据：** 在当前 Loop 的指证中使用了前序 Loop 获得的证据。
- 这类情况在 state 中通常以注释标注（`L{N}已获得`、`L{N}获得`）
- 可视化时应显示证据获取 Loop，而不只是 ID

### 3.4 NPC 心态递进的可视化建议（从 mindset 字段推断）

`mindset` 字段是单一字符串，描述的是 NPC 在**整个 Loop 的指证过程中**的心理基调，而不是每轮的状态。

可视化器可以用以下方式处理：

**方式 A：整体基调标注**
把 mindset 字符串显示在指证块的顶部，作为读懂每轮 NPC 反应的背景色。
不要把 mindset 切分成每轮状态——设计者没有为每轮写单独的心态。

**方式 B：从 result 推断递进**
每轮的 `result` 文本隐含了心态变化（防守 → 动摇 → 崩溃）。可视化器可以提取 result 的关键动词/行为来标注每轮的状态：

```
R1 result 关键词：「慌张辩解」→ 状态：防守，谎言未破
R2 result 关键词：「部分承认...但坚称」→ 状态：动摇，升级谎言
R3 result 关键词：「Rosa 最终供出」→ 状态：崩口，核心坦白
```

这种推断不需要 AI，只需要从 result 文本中查找以下关键短语模式：
- "无法解释"、"慌张辩解"、"坚称" → 防守
- "部分承认"、"说漏嘴"、"退守" → 动摇
- "承认"、"含泪"、"供出"、"坦白" → 崩口

**方式 C：mindset 字段的段落模式**
如果 mindset 字段包含破折号（`——`），破折号前通常是表面状态，破折号后是内心状态。可以拆分显示：
- 表面：「越来越动摇」
- 内心：「每一轮证据瓦解她的谎言，玩家点破 Morrison 不知情后终于开口」

## 第四部分：NPC 知识边界的可视化建议

### 4.1 三种知识状态

State 文件中每个 NPC 有三个知识相关字段：

```yaml
knows:       # 知道的内容（真实存在于脑中）
does_not_know:  # 真的不知道
lie:         # 知道但在骗你
```

可视化时应区分三种状态，但不能等权显示——这三个字段对应了三种**完全不同的叙事含义**。

### 4.2 视觉区分建议

**knows（真实知道）：**
- 这是 NPC 的知识储备，不直接显示给玩家（会破坏游戏体验）
- 在可视化器中建议用**折叠展示**：策划可以展开查看，但默认不展开
- 颜色建议：中性色（灰色）

**does_not_know（真的不知道）：**
- 对叙事设计很重要：这是 NPC 无法提供信息的边界，也是"问了也没用"的范围
- 建议用**浅色/细字体**展示，视觉上比 knows 更"轻"
- 重点标注：如果 `does_not_know` 中的某条内容恰好是当前 Loop 玩家最想知道的方向（可与 `player_context.goals` 对比），加注「玩家死角」提示策划

**lie（知道但在骗）：**
- 这是 NPC 最关键的叙事张力所在
- 建议用**高对比度展示**，但要用"折叠+权限"处理：
  - 策划视图：显示 `lie.content`（谎言内容）和 `lie.truth`（真相）的对比
  - 默认视图：只显示"此 NPC 有谎言"的标记，不展开内容

**NULL lie（诚实的 NPC）：**
- lie == null 说明 NPC 在当前 Loop 没有主动欺骗
- 不要标注"诚实"——这个标签本身就是对玩家的剧透
- 在策划视图中可以显示 `lie: null` 的状态，提示策划这个 NPC 是透明信息来源

### 4.3 「知道但在骗你」vs「真的不知道」的视觉对比

这是最核心的可视化问题。建议用以下方案：

**方案 A：双列对比（适合详细视图）**

```
[NPC名称]  [is_liar: true / false]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
知道的           |  真的不知道
─────────────────|───────────────
• 自己当晚在走廊  | • Webb 谋杀的完整真相
• Morrison 迷晕  | • Morrison 的最终计划
  Zack 全过程    |
• 事后捡走毛巾   |
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
谎言内容："从 9 点半开始在地下室打扫"
真    相：[折叠，点击展开]
```

**方案 B：单列带标签（适合紧凑视图）**

在 knows 列表中，对于 lie 中出现的相关条目加"[隐瞒]"标签：
```
• 自己当晚在后台走廊工作（非地下室） [隐瞒]
• 目睹 Morrison 迷晕 Zack 全过程     [隐瞒]
• 事后自行捡走毛巾                   [隐瞒]
• Morrison 不知道她在场
• Morrison 是危险的腐败警察
```

这种方式的优势：玩家死角一目了然（哪些事 Rosa 知道但没说），而不是把 knows / lie 分成两个不相关的列表。

### 4.4 特殊情况处理

**NPC 没有独立 lie 字段，但 is_liar == true：**
这种情况下，谎言可能体现在 `testimony_ids` 对应的证词内容中（通常有 `⚠谎言:` 标记）。可视化器应从 testimony_ids 的注释中提取谎言内容，或在 NPC 条目旁标注"谎言来源：见证词 [ID]"。

**NPC 的 `does_not_know` 中出现了后续 Loop 才揭晓的信息：**
这是叙事一致性检查的核心。比如 L4 开篇 Emma 的 `does_not_know` 中包含"Helen 当晚在楼顶"——而这正是 L4 指证要揭示的内容。可视化器可以对这类条目加"本 Loop 将揭示"的标记，帮助策划快速验证 NPC 知识边界设计是否自洽。

具体判断方式：将 `does_not_know` 中的每条内容与 `post_expose_knowledge` 中的条目做文本模糊匹配，如果有重叠，标注"本 Loop 揭示"。

**指证目标 NPC 在探索阶段和指证阶段分别出现：**
State 文件中有时同一个 NPC 在 `scenes[].npcs` 和 `expose` 中都有条目（如 Rosa 在 `SC1110` 和 `expose.rosa` 中各出现一次）。可视化器应将这两个条目合并展示，以"探索阶段对话"和"指证阶段"为标签区分，避免显示为两个不同的 NPC 实例。
