# 对白撰写 Skill

产出经 `sync_to_json.py` 进 Unity 工程的**正式中文对白**。目标只有一句：

> **写出像真人一来一回说话、角色声纹分明、为整章叙事服务、推理严密、不提前破梗的对白草稿。**

不是"用最少字传达信息点"，不是"NPC 演讲 + Zack 点头"。

## 适用 / 不适用

- 适用：state 已定稿（`/team-loop` 或 `/unit-state-generator`），需整 Loop / 整 Unit 的完整对白（Talk + Expose + 过场）。
- 不适用：只改单 NPC 或小修文案 → 用 `/team-design`。

## 输入约定

| 必需 | 说明 |
|------|------|
| state 路径 | 单个 `loop{N}_state.yaml` 或含多个 state 的目录 |
| Unit 大纲 | `剧情设计/Unit{N}/Unit{N}_大纲.md`（**写手必读，见 R2**） |
| 人物设计目录 | `剧情设计/Unit{N}/Characters/` 下涉及 NPC 的 `{npc}.md`（**角色声音的唯一依据，见 R3**） |
| 全局剧情目录 | `剧情设计/` 下全局主线、世界观、主角设计、各 Unit 大纲/主线推进/state（**全局关系与知识账本依据，见 R14**） |

解析顺序：argument 给路径→按给的用；仅给 state→同级/上级找 `Characters/` 与 `Unit{N}_大纲.md`；找不到→AskUserQuestion 问，不假设默认路径。

**本 skill 不引用任何外部"对白语言风格资料"**；但必须读取项目内已手改成稿做"文笔校准"。默认参考：
- `AVG/对话配置工作及草稿/Unit2/Loop*_生成草稿.md`：项目成稿口感基准
- `AVG/对话配置工作及草稿/Unit9/Loop*_生成草稿.md`：当前重构章节的手改校准

只学习写法，不继承事实：可以提炼句长、节奏、动作括注、证据压迫方式、结尾留勾；禁止搬运 Unit2 的剧情、证据、设定、NPC 关系或原句。角色声音仍以 `{npc}.md` 为唯一依据。

知识池 `npc_knowledge_pools.yaml` 不假设存在；NPC 知识边界以 state 内嵌 `active_topics` / `withheld_topics` 为准。

## 文本规则 R1–R14（贯穿全流程，必须原样粘进 Phase 2a/2b/4/5 的 agent prompt；违反视为 FAIL）

### R1 · 一来一回的对白，不是演讲

对白是 dialogue 不是 monologue。任何一方都不能被写成"一方播报/演讲 + 另一方点头/转述"。该追问就追问、该回呛就回呛，可打断、可半句、可语气词、可带情绪。沉默用动作括注承载，不写空台词。一段读下来要像两个人在说话，不像一个人在做报告。

### R2 · 先懂大纲，再写本场

写手动笔前必须读 Unit 大纲，弄清这段对话**在整章里的叙事功能**：它揭示哪一层真相、为后面哪个 Loop 埋什么伏笔、玩家此刻该有的情绪、它在角色弧光的哪一步、它和上下场如何衔接。对白为这个功能服务，不孤立写一场戏。Phase 2a 讨论与 Phase 2b prompt 必须显式写出"本场叙事功能"。

### R3 · 人设是角色声音的唯一依据

角色怎么说话，只看 `{npc}.md` 的"声音风格 / 禁忌 / 设计备注 / 关键关系"。照人设写，不照任何外部语言风格模板、不堆刻板族裔/阶级标签。每个 NPC 首句要让人不看档案也认得出是谁。人设里写好的口头禅、反讽标记、禁忌、设计核心是**绑定约束**，不是参考。

### R4 · 人设质感与连接戏（正向强制）

state 信息点是骨架，不是全部。每个单元必须有服务 ① 人物声音 ② 人物关系 ③ 世界/时代质感 的台词；碎嘴型人设的废话**本身就是人设**，要写足。
判据：一句非 state 信息点的台词，服务人物/关系/世界任一 → 要；与剧情、人物、世界全都无关的纯空镜渲染（"夜色深沉，这城市从不眠…"）→ 禁。
自检：删掉所有 state 信息点后，剩下台词仍能让人认出"这是谁 / 什么关系 / 什么世界"——认不出 = 血肉不足，打回。

### R5 ·（已删除——由 AGENTS.md 项目指令全局管辖）

> 原 R5（悬疑感·游戏设计原则一）已删除：**不替玩家下结论 / 不说"所以这说明…" / Talk 阶段不替玩家戳穿谎言 / 藏梗>明示 / 后续 Loop 才揭示的信息当前 Loop 必须隐藏或伪装 / NPC 谎言退守是自然反应非 Zack 喂话**——这些是 AGENTS.md「游戏设计三大原则·原则一·悬疑感优先」，**全局生效、高于本 skill 所有规则**，本 skill 不再重述但全程适用。R9/R10 及下文凡提"原则一/破梗"即指此。编号 R5 保留为占位，不重新编号（R6–R10 引用不变）。
>
> **Zack 声音说明（保留，非原则一，属 R1/R10 territory）**：Zack 可以激烈地一来一回、追问、回呛、带情绪、用私人重量（"我妈"）顶上去——他逼的是线索，不是替玩家说出答案。本 skill 不设"Zack 钝 / 句数上限 / 零感叹号"——那套已废弃。

### R6 · 信息严密 · 游戏设计原则二

证据物理属性 / 数值 / 时间前后一致、跨 Loop 引用不矛盾；证词摘要与原文信息密度对等（不丢关键限定词，不膨胀出未述结论）；每个 NPC 在其知识边界内自洽——即使说谎，谎言本身也有内在逻辑。

### R7 · 旁白克制不空 + 无大段独白

环境/动作旁白少写；世界质感由**人物台词**承载，不靠旁白渲染。任何角色都不写大段内心独白 / 演讲。纯画面 / 转场 / NPC 静默反应用 `<!-- -->` 注释或并进相邻台词的动作括号 `[...]`，**不写成空 `**角色名**` 发言块**（承载 `@get`/`@branch`/`@lie`/`@goto` 机制处除外，用 @指令显式标记）。

### R8 · 格式与命名

- 节点边界 = 一个 `**角色名**` 发言块（其下连续 `>` 行属同一节点）；不写 9 位对话 ID（后置 `assign_ids.py` 补）。
- 分支 / 提取 / 指证一律 `@指令`，语法详见 `AVG/对话配置工作及草稿/无ID对话语法规范.md`（**这是语法规范，保留必读；不是风格资料**）。speed-ref：
  ```
  节点 **角色名** [动作] + 下方 > 台词行
  画面 <!-- 注释 -->（不成节点）
  分支 @branch 名 / @opt "文本" -> 标签 #维度 / @path 标签 / @goto 标签 / @label 标签
  提取 @get 证词|证据 {配置ID} "{摘要}" #{keyInfoType}
  指证 @round N / @lie anchor={testimony_id} / @evidence "名" -> 标签 #correct|#trap "理由"
  ```
- 角色发言名全程统一中文译名（如 扎克·布伦南 / 莫里森 / 艾玛·奥马利），不中英混用。
- **JSON 文件名一律取自 state 字段原值**——见下「命名规则」。不自创 `{npc}_{conv}.json`。

### R9 · 承上启下：开场锚定 + 结尾留勾（都不机械）

每段对话不能让玩家"像从中途开始看"，也不能戛然而止没有去向。

- **开场锚定（≠重新进场）**：每段非硬切对话，开头一两句自然台词/动作让玩家落地"我此刻在哪、上一步刚发生了什么、现在的紧张点是什么"。靠人物之口/动作带出，**禁旁白 info-dump、禁重复进场（推门 / 自我介绍 / 重新宣告已知结论）**。判据：一个中途进来的玩家读前两个发言块，能不能知道现在是什么处境。
- **结尾留勾（≠任务提示）**：结尾由人物自然带出"下一步往哪使劲 / 还有什么没解决"。**禁 Zack 总结式"我们现在该去 X 做 Y"、禁任务播报、禁破梗（不说该找什么证据证明什么——那归 AGENTS.md 原则一）**。承载者：Emma 行动导向的一句或一个动作、Zack 钝但有指向的短句、或 NPC 撂下的话本身成钩（如"拿硬证据我签搜查令"）。判据：读完最后两个发言块，玩家心里有没有"接下来该往哪动"——且这念头是被勾出来的，不是被告知的。
- 硬切 cutscene 不强制开场锚定（它本身就是建立场景），但结尾仍要有自然的前推 / 悬念。
- **段与段衔接**：上一段的"勾" ≈ 下一段开场要落地的"处境"（如 §3 Emma"我去打个电话"勾 → §4 开场落地"刚打完 Foster 的电话"）。Phase 3 通读缝合负责打通这条链。

### R10 · 地位跷跷板：场景必须转弯（不机械、不破梗）

戏剧依据：Keith Johnstone 状态理论 + McKee "场景必须转弯"。R1 只查"有没有来回"，R10 查"来回里有没有权力的此消彼长"。

- **上风至少易手一次**：每场对手戏（审讯 / 对质 / 谈判 / Expose / 有张力的搭档戏），"谁占上风"必须在场内**至少易手一次**，且每次易手是被一句台词或一个动作**挣来的**（一个"转弯"），不是设定好的、不是旁白宣布的。从头到尾一方稳压另一方 = 死场，打回。
- **低地位角色也要赢到至少一拍**：弱势/惊惶/说谎的一方，必须有至少一拍用示弱、装傻、绕开、踩痛处等方式**真的把强势方挡回去一次**（强势方要可见地"得换招"）。低位人设 ≠ 全程挨打。
- **第三方支点（尤其 Expose）**：可引入在场第三方（如 Morrison）作为搅动天平的支点——他倒向谁，上风就偏谁。鼓励用，这是 status play 的高级形态。
- **不机械、不破梗**：易手靠**态势**（谁逼住谁 / 谁挡回谁 / 谁失足 / 谁被第三方掀翻），不靠说破谜底、不靠"我赢了"式自陈、不靠旁白标注"地位回升"——让玩家从台词动作里读出权力流向。与 AGENTS.md 原则一并存：转弯是势的翻转，不是结论的泄露。
- **判据**：把整场画成"上风曲线"——有没有翻折点？翻折是不是被某句话/动作挣的？低位角色有没有赢过一拍？一条直线（一方恒高）= FAIL。
- **豁免**：硬切 cutscene（本身建立场景）、纯过场无对手关系的不适用；凡有两方张力的场都适用。
- **正面样本**：`AVG/对话配置工作及草稿/Unit9/Loop1_生成草稿.md` §5 Rosa 指证（903901001–903901021）——Rosa 抢攻→Zack 夺回→玩家出 9711 Zack 压下→Morrison 横插"非法搜证"把 Zack 也掀翻→Morrison 动摇偏回 Zack→Rosa"枪的问题被挡回来了"地位回升→Morrison 倒向 Rosa 压住 Zack。上风易手五次以上 + Morrison 第三方支点，照这个写。
- **反面样本**：同文件 §3 Rosa 审讯（903001001–903001402）——Zack 全程高位、Rosa 全程缩，地位一次没易手 = 死场。修法：903001105→903001106 应让 Rosa 的"我只知道我看到的"真的把 Zack 挡回一拍（Zack 可见地得换角度问），而非 Rosa 继续缩、Zack 继续稳压。

### R11 · 指证是玩家情绪高点，必须有紧张感和击破感

Expose / 指证不是普通问答，也不是证据讲解。它是玩家用前面积累的证据主动撬开谎言的情绪高点，必须写出"压上去 → 被挡回 → 再压上去 → 击穿"的紧张感。

- Zack 出证不能平铺直叙，必须带明确的压迫动作或语气变化：把证据放上桌、推近、停顿、逼问、抓住对方退守漏洞。
- NPC 不能只是"啊我错了"，必须有更重的情绪反应：抢攻、狡辩、反咬、失控、沉默、身体退缩、寻找第三方支点。
- 正确证据命中时，要有可感知的击破瞬间：对方话术断裂、上风易手、旁观者态度变化、现场空气收紧。
- 陷阱证据也不能平淡，必须让 NPC 或第三方把玩家挡回来，让玩家明白"这个证据为什么打不穿"。
- 禁止把指证写成推理 PPT：Zack 不能连续解释一整串逻辑，NPC 不能只负责点头崩溃。每一轮都要有对抗、退守和二次推进。

### R12 · 角色声纹与冲突姿态：台词不能换人说

每个角色必须从自己的身份、欲望、恐惧、关系和防守动作里说话。同一条信息由不同角色说出时，句式、攻击角度、回避方式、情绪泄漏都必须不同。

每个 speaking character 在进入 writer 前必须有"角色声纹卡"：
- 本场欲望：他想从 Zack / Emma / 现场拿到什么。
- 本场防守：他最不想承认、暴露或被逼近什么。
- 说话武器：身份、专业术语、礼貌、讽刺、沉默、威胁、装糊涂、讨好、转移话题等。
- 句式节奏：短句/长句、反问密度、是否绕弯、是否过度解释、是否带口头禅。
- 情绪泄漏：被逼到时会急、冷、讥讽、讨好、失控，还是突然安静。
- 关系温度：他对 Zack / Emma / 其他 NPC 的说法有什么差别。
- 与本场其他角色的差异：同说一条事实，他和别人最不一样的角度是什么。

硬判据：
- 遮住角色名，仍应能凭语气和姿态大致认出是谁。
- 把两名角色的台词互换后仍然成立 = FAIL。
- 所有人都用同一种"解释事实 / 否认 / 反问 / 情绪爆发"推进信息 = FAIL。

### R13 · Talk 必须是开篇问候情境 + 1-3 个选项

普通 Talk 不是审讯清单。每段 Talk 必须先有自然的开篇问候/试探/当下情境，再进入 `@branch` 的 1-3 个玩家选项。每个选项负责不同证词或信息维度，不能多个选项换句话问同一件事。

- 开篇先建立 Zack 与 NPC 此刻的见面情境、关系温度和场景压力，不直接抛证词问题。
- 选项数 1-3 个；每个选项对应一个清晰的信息维度或 `@get` 目标。
- 选项文案必须是玩家此刻合理想问的话，不是设计师的信息点标题。
- Zack 只能问他此刻已经看到、听到、读到、被告知、或有合理现场依据能怀疑的事。
- 如果 Zack 还不知道某个人名、地点、物件、时间点或结论，不能直接点名发问，只能从更宽的问题切入。
- 选项内仍要一来一回、有攻防、有角色反应；不能点选后 NPC 直接交付证词。

硬判据：
- 没有开篇情境直接进入分支 = FAIL。
- 超过 3 个普通 Talk 选项，或选项信息维度重复 = FAIL。
- Zack 直接问出当前知识账本里没有来源的信息 = FAIL。

### R14 · 全局上下文先于局部对白：不能只照 state 写

state 是配置骨架，不是完整剧本。写每一段对白前，必须理解角色在全局故事中的位置：
- 他和 Zack / Emma 是初见、熟人、电话联系过，还是只存在档案里。
- 他在主线里背着什么压力、欠着谁、怕谁、误会谁。
- 他此刻知道什么、不知道什么、以为自己知道什么。
- 当前 Loop 只能写到哪一层，后续 Unit / Loop 的信息只能埋气味，不能收答案。

如果 state 没写清，但全篇 state / 大纲 / 主线文档能推断出来，必须按全局理解修正对白。例：如果全局关系显示某角色与 Zack 第一次面对面见面、但 U2 有过电话沟通，台词要同时承认"电话联系过"和"第一次见面"的陌生感，不能写成熟人寒暄。

如果全局文档和当前 state 冲突，不能擅自决定，列为"需用户裁决"。

## 全局上下文包与三张卡

每次生成对白前，lead 必须建立"全局上下文包"。用户不怕耗时/耗 token 时，宁可多读文档，也不要只照目标 Loop state 写。

### 必读范围

- 当前 Unit：全部 `state/loop*_state.yaml`、Unit 大纲、主线推进、人物总览、涉及场景文档、涉及 NPC 人设。
- 全局主线：`剧情设计/00_世界观与角色/主线故事全貌.md`、世界观、Zack / Emma / Mickey 等主角设计。
- 跨 Unit 关系：`剧情设计/Unit1` 到 `剧情设计/Unit5` 下可发现的"大纲 / 主线推进 / state / 人物设定"摘要，用于判断关系状态、旧联系、电话沟通、主线压力和长期伏笔。

### 使用边界

- 全局文档用于理解关系、知识边界、主线压力和伏笔层级；不能把后续真相当作当前 Loop 已知事实。
- Unit9 / Unit10 仍按 AGENTS.md 当作独立新章节；旧 Unit1 / Unit2 只能在用户明确要求或当前文档无法解释时作参考，并必须标注"参考旧版"。
- 全局补笔只能补人物关系、世界质感、主线压力和情绪余波；不能新增证据、改写指证、替玩家下结论。

### 输出卡

Phase 1 前必须输出并传给后续所有 agent：
- **全局人物状态卡**：每个 speaking character 与 Zack / Emma / 关键 NPC 的关系状态（初见/见过/电话联系/听说/敌对/合作）、当前压力、长期隐瞒、不能提前说破的信息。
- **Zack 知识账本**：到本段开场为止 Zack 已知的信息、来源、可问法、禁止直接问法。
- **全局补笔清单**：本段可以适当加入的关系/主线/世界细节，以及必须避开的后续真相。

## 项目文笔校准

每次生成对白前，lead 必须先从项目内手改稿提炼一张"文笔校准卡"，传给后续所有 agent。校准卡不是风格模板，不允许抄句；它只规定本项目成稿的口感、节奏和动作密度。

### 样本选择优先级

1. 目标 Unit 已有手改稿中，同 NPC / 同场景类型 / 同功能的片段。
2. `AVG/对话配置工作及草稿/Unit9/Loop*_生成草稿.md` 中同类片段（当前重构章节口感）。
3. `AVG/对话配置工作及草稿/Unit2/Loop*_生成草稿.md` 中同类片段（项目成熟成稿口感）。

同类型优先于同角色：Opening 对 Opening，普通 Talk 对普通 Talk，Expose 对 Expose，post_expose 对 post_expose。若样本很多，每类抽 1-2 段即可，不要把整章塞进 prompt。

### 校准卡格式

输出 5-8 条可执行观察，每条都要能指导 writer 改写。例如：
- 台词短促，一句只推进一个动作、一个情绪拍或一个压力点。
- 证据先作为物件落到场内，再让 NPC 反应；不要先讲成推理 PPT。
- 动作括注服务攻防、关系和状态变化，不做空镜渲染。
- 角色会绕开、顶回、嘴硬、停顿；不能像按表交付信息点。
- 结尾用人物自然抛钩，不用任务播报。

禁止写成"模仿 Unit2 某段"或贴大段样本文字；禁止迁移 Unit2/旧 Unit 的剧情事实、证据关系、NPC 关系、原句或口癖。

## 命名规则（文件名来源 = state，不自创）

| 段 | section 头文件名取自 state |
|---|---|
| Opening | `opening.talk` 值 → `## Talk: {值}.json` |
| Talk | `scenes[].npcs.<key>.talk` 值 → `## Talk: {值}.json` |
| Expose | `expose.target_talk` 值 → `## Expose: {值}.json` |
| post_expose | `post_expose.talk` 值 → `## Talk: {值}.json` |

- 场景分组头（人类可读）：`## §N. Scene {scene_id} — {描述}`，与文件名解耦。
- **不存在"没有对话名字的内容"**：每段对白必属于 state 某个 `talk` / `target_talk` 字段。纯过场（如结尾电报）并入语义上最贴近的、state 已命名的相邻对话段（通常是其前一段 post_expose / Talk 的 talk 文件），**不另起无名 cutscene 文件**。
- 若遇到 state 某段确实查无对应 `talk` 字段且无法判断归入哪段 → 视为 state 缺陷，lead 用 AskUserQuestion 停下来问用户归属，**绝不自创文件名**。

## Pipeline

### Phase 0 · 前置读取（lead 自己做，不 spawn agent）

读齐并汇总成"上下文包"：
- state（用户指定文件或目录下全部 loop state）；如果只给单个 loop，也要读取当前 Unit 其余 loop state 做全局理解
- **Unit 大纲全文**——lead 提炼出"本 Loop / 每段对话在整章里的叙事功能"，作为 R2 的依据传给后续所有 agent
- **全局上下文包**——按上文读取当前 Unit 全部 state/大纲/主线推进/场景/人物、`剧情设计/00_世界观与角色/`、以及 Unit1-Unit5 的大纲/主线推进/state/人物摘要，输出"全局人物状态卡 / Zack 知识账本 / 全局补笔清单"（R14）
- 涉及 NPC 的全部 `{npc}.md`（声音 / 禁忌 / 设计备注）
- `AVG/对话配置工作及草稿/AVG对话配置规则.md` + `.Codex/rules/dialogue.md` + `AVG/对话配置工作及草稿/无ID对话语法规范.md`
- 文笔校准样本：按"项目文笔校准"读取 Unit9 / Unit2 手改稿片段，输出"文笔校准卡"。若目标 Unit 不是 Unit9，也仍可用 Unit2 作项目成稿基准，但不得继承 Unit2 事实

任一必需文件缺失 → AskUserQuestion 停。

### Phase 1 · 派发方案确认

读 state + 大纲后提取并输出给用户确认：
- 讨论单元清单（每 NPC Talk / 每 Expose / 每过场），每单元标注 **本场叙事功能（取自大纲，R2）**
- 每单元的 **JSON 文件名（取自 state 字段，R8/命名规则）** 与 @branch/@get/@lie 清单
- 同场景 / 连续场景的衔接关系（防各写手重复进场）
- 平行场景隔离与破梗红线（本 Loop 必须隐藏什么）
- 全局人物状态卡：每个 speaking character 与 Zack / Emma / 关键 NPC 的关系状态、当前压力、长期隐瞒、不能提前说破的信息
- Zack 知识账本：每段 Talk 开场前 Zack 已知什么、来源是什么、能怎样问、不能直接问什么
- 全局补笔清单：每段可加入的关系/主线/世界细节，及必须避开的后续真相
- 文笔校准卡：5-8 条本次写作必须遵守的成稿口感观察，明确"只学写法，不继承事实"

AskUserQuestion：方案 OK → Phase 2；调整 → 改；取消。

### Phase 2a · 两策划并行讨论（每单元一对）

> 注：这里是【维度分工产出】——叙事策划做声音/上风曲线，推理策划做分支/破梗，两者做的是互补的两半，不是对同一决策的两种意见，更不是圆桌辩论。仅当两者就【同一个决策】真正冲突（非互补）时，才按"信息严密性>叙事质感"优先级合并，严重冲突 AskUserQuestion 交用户。

每单元并行 spawn 一对 narrative-designer + puzzle-designer。两者 prompt 都必带：state 对应段落**原文**（不压缩）、`{npc}.md` 路径、**大纲中本场叙事功能**、全局人物状态卡、Zack 知识账本、全局补笔清单、文笔校准卡、文本规则 R1–R14 原样、破梗红线。

narrative-designer 输出：① 每段叙事目标（含本场在整章的作用，R2）② **角色声纹卡**（R12：每名 speaking character 的欲望/防守/说话武器/句式节奏/情绪泄漏/关系温度/与他人差异）③ **冲突姿态卡**（谁用什么逼近，谁用什么挡回，差异来自人物视角而非信息点）④ 一来一回的对话动线设计（R1：谁追问谁、在哪顶、在哪退）⑤ 情绪节拍 ⑥ 禁忌清单 ⑦ **同信息差异样张**：同一条事实至少用两名角色口吻各写 1 句，遮名可辨；另给 2-3 句声音样张，标注"writer 照这个口语温度和密度写，不是照 bullet 缩写" ⑧ **上风曲线设计（R10）**：本场地位起点、谁在哪拍翻转、靠什么台词/动作翻、低位角色赢哪一拍、（Expose）第三方支点怎么用 ⑨ **全局关系落地**：本单元如何落实"初见/熟人/电话联系/听说"等关系状态，哪些全局补笔可用，哪些后续真相禁用 ⑩ **文笔校准落地**：本单元如何落实校准卡的 2-3 条关键观察。

puzzle-designer 输出：① **Talk 结构设计**（R13：开篇问候情境 + 1-3 个选项；每个选项对应不同证词/信息维度）② **Zack 问题依据表**（每个选项/追问来自哪条已知信息；禁止直接问什么）③ 分支维度设计 ④ 信息点分布（每场景 1-3 个 state 信息点）⑤ @get 时机（间隔 3-5 节点）⑥ Expose 轮次/谎言/证据/陷阱逻辑 ⑦ 破梗清单 ⑧ **指证高点设计（Expose 必填）**：每轮正确证据的击破瞬间、NPC 退守/反咬、陷阱证据如何挡回玩家。

lead 合并两份为"统一方案"，冲突原则：**信息严密性(R6/R13/R14) > 叙事质感(R1/R3/R4/R12) > 逻辑闭环**；推理与叙事冲突时以推理为准、让叙事换表达；严重冲突 AskUserQuestion 交用户。**合并必须保留并整合角色声纹卡、冲突姿态卡、Talk 结构设计、Zack 问题依据表、声音样张**，连同本场叙事功能一起传给 writer。

### Phase 2b · dialogue-writer 按方案执笔

每单元 spawn 一个 `dialogue-writer`，prompt 必含：
- 统一方案 + **角色声纹卡 / 冲突姿态卡 / 同信息差异样张 / 声音样张原样**（注明"目标是这个口语密度与人设质感，state 信息点之外按 R4 主动补人设闲聊/世界质感/关系温度"）
- **Talk 结构设计 + Zack 问题依据表原样**（R13：开篇问候情境 + 1-3 选项；每个问题必须有当前知识来源）
- **全局人物状态卡 / Zack 知识账本 / 全局补笔清单原样**（R14：关系状态、主线压力、后续真相边界必须写对）
- **文笔校准卡原样**（注明"只学习节奏、动作密度、证据落场方式和结尾留勾，不许继承样本剧情事实或原句"）
- **大纲中本场叙事功能**（R2）
- state 对应段落**原文**（不摘要）
- `{npc}.md` 路径，并把其"声音风格"整段示例原样粘进 prompt（R3）
- 文本规则 R1–R14 原样粘贴
- 场景连续性提示 + 平行场景隔离清单
- 目标 JSON 文件名（取自 state 字段，R8/命名规则；不分配 ID 段）
- `无ID对话语法规范.md` 路径 + speed-ref 原样
- 产出路径：临时文件 `AVG/对话配置工作及草稿/{Unit}/.temp_{state文件名值}.md`

writer 动笔前必须做两项不进入正文的自检：① **换名测试预演**：任选同场两名角色，同一信息各写一句，确认不可互换；② **Zack 知识边界预演**：逐个 Talk 选项标出 Zack 的已知来源，没有来源的改成更宽的问题。

多 writer 并发（不同临时文件）。

### Phase 3 · 合并与缝合

lead 负责：
1. 按场景顺序拼入主文件 `AVG/对话配置工作及草稿/{Unit}/Loop{N}_生成草稿.md`
2. section 头：文件名取自 state（R8/命名规则）；场景分组头人类可读
3. 纯过场并入语义最贴的已命名对话段（命名规则）
4. 清理 `.temp_*.md`
5. **@ 指令闭合校验**：每 `@opt` 有 `@path` / 每 `@path` 以 `@goto` 收束 / `@goto` 目标已定义 / `@branch` ≥2 / 无重复标签 / `@get` 证词7位·证据4位 / 每 `@round` ≥1 `#correct` / R1 `@lie anchor` = Talk 已收集证词（详 无ID规范 §6）
6. **通读缝合**：按 voice 与 R1 通读全稿，补单元间连接句、消除"信息点清单感"和并行写作接缝；按文笔校准卡自检"是否像项目手改成稿，而不是 state 扩写 / 策划说明 / 推理 PPT"；按 R4 自检"删掉信息点后是否仍认得出人物/关系/世界"，按 R1 自检"是否有一方在演讲/另一方在点头"，**按 R9 逐段自检"中途进来读前两块知不知道处境 / 读完后两块知不知道往哪动，且都不机械"，并打通段与段衔接（上一段的勾 ≈ 下一段开场要落地的处境）**，**按 R10 给每场对手戏画"上风曲线"自检：有无被挣得的翻折、低位有无赢拍、（Expose）第三方支点是否用上——一条直线则回补转弯**，**按 R11 检查每个 Expose 是否有紧张感与击破感：正确证据命中有击破瞬间，陷阱证据能有力挡回，NPC 有退守/反咬/失控**，**按 R12 做遮名/换名测试：主要角色各抽 3 句，遮名能否认出；互换后仍成立则回补声纹**，**按 R13 检查每个普通 Talk 是否有开篇情境 + 1-3 选项 + 不同证词维度 + Zack 问题依据**，**按 R14 检查初见/旧联系/电话沟通/主线压力是否符合全局上下文包**，不足处回补
7. 主文件路径呈给用户

### Phase 4 · 并行审查

并发 spawn，每审查员 prompt 必带：草稿路径、state 路径、大纲路径、全局人物状态卡、Zack 知识账本、全局补笔清单、文笔校准卡、文本规则 R1–R14。
- `dialogue-reviewer` ×每 Loop 一份，含以下**强制闸门项**（与破梗同级，可 FAIL）：
  - **【活人测试】** 随机抽 3 段：像不像这个身份的人在开口说话？读起来像"信息点清单/舞台提示" → FAIL。
  - **【成稿感测试】** 随机抽 3 段，对照文笔校准卡：是否像 Unit2 / Unit9 手改稿那种"短句推进、动作带压力、证据落场、自然留钩"的项目成稿口感？如果读起来像 state 信息点扩写、策划说明、推理 PPT，或套用 Unit2 原剧情事实 → FAIL。
  - **【一来一回测试】** 抽查对白：是不是一方在演讲/转述、另一方在点头？是 → FAIL。Expose 同查（Zack 出证是否在跟 NPC 顶，而非单方面摆证据）。
  - **【承上启下测试】** 随机抽 2 段：中途进来读前两个发言块知不知道处境？读完最后两个发言块知不知道往哪动？这两处是否机械（旁白 info-dump / 重新进场 / 任务播报 / Zack 总结式）？缺失或机械 → FAIL（R9）。
  - **【跷跷板测试】** 抽对手戏画"上风曲线"：有没有被某句话/动作挣得的翻折点？低位角色有没有赢过一拍？一条直线（一方恒高）或翻折无挣得 → FAIL（R10）。Expose 另查第三方支点是否用上。
  - **【指证高点测试】** 抽查每个 Expose：正确证据命中时有没有"击破瞬间"？NPC 有没有反咬/退守/失控等强反应？陷阱证据有没有被有力挡回？如果读起来像普通问答、证据讲解、平静复盘、推理 PPT → FAIL（R11）。
  - **【声纹互换测试】** 主要 speaking character 各抽 3 句，遮住角色名是否能认出；任意两名角色台词互换仍成立，或所有人同一种否认/解释/反问方式 → FAIL（R12）。
  - **【Talk 选项分工测试】** 每个普通 Talk 是否有开篇问候情境 + 1-3 个选项；每个选项是否负责不同证词/信息维度；选项是否像玩家会问的话而非设计标题；不满足 → FAIL（R13）。
  - **【Zack 知识边界测试】** 逐个 Talk 选项和追问核对 Zack 知识账本：每个问题是否有当前已知来源；若直接问出未获得的人名、地点、时间、物件或结论 → FAIL（R13/R14）。
  - **【全局连续性测试】** 检查角色关系是否符合全局文档：不能把第一次见面写成老熟人，不能把电话联系过的人写成完全陌生，不能引用后续 Unit / Loop 真相；违反 → FAIL（R14）。
  - 另查 AGENTS.md 原则一（是否替玩家下结论/破梗/Talk 戳穿）、R2（是否贴合大纲叙事功能）、R3（人设巧思/禁忌是否兑现）、R4（删信息点后是否认得出人物/关系/世界）、R7（空发言块/大段独白）、R8（命名是否取自 state）、R9（段间衔接是否打通）、R10（上风曲线是否有翻折）、R11（指证是否有情绪高点与击破感）、R12（角色不可互换）、R13（Talk 结构）、R14（全局上下文）。
- `voice-print-auditor` ×主要 speaking NPC（每名 NPC 一份）：集中阅读该 NPC 在本次草稿中的所有台词 + `{npc}.md` + 角色声纹卡，审查声纹漂移、AI 雷同句式、现代腔、与关系状态不符的熟悉度；P1 视为可打回项。
- `consistency-checker` ×1：@get/@lie/@evidence ID 与 state 一致、证据物理属性/证词密度(R6)、角色名统一、命名取自 state、@闭合。
- `timeline-auditor` ×1：跨 Loop 信息泄露、伏笔只种不收、证据/证词与疑点解锁时序、L{N}→L{N+1} handoff。

player-simulator 不默认跑（整 Unit 就绪后走 `/playthrough-audit`）。

### Phase 5 · 内容总监闸门

spawn `content-director`，输入 Phase 4 全部审查报告全文 + Phase 1 方案 + 全局上下文三张卡 + 用户原始要求。输出：综合评分表（每 Loop PASS/WARNING/FAIL）、FAIL 必修清单(P0/P1)、建议改(P1/P2)、需用户裁决的架构问题、下一步。
**质感是可打回闸门**：活人测试 / 成稿感测试 / 一来一回测试（R1）/ 承上启下测试（R9）/ 跷跷板测试（R10）/ 指证高点测试（R11）/ 声纹互换测试（R12）/ Talk 选项分工与 Zack 知识边界测试（R13）/ 全局连续性测试（R14）或 R4 血肉自检 FAIL → 必返修，不得只标 WARNING 放行；"像信息点清单 / 像演讲 / 像从中途开始看 / 戛然而止 / 一方恒压（场景不转弯）/ 角色台词可互换 / Talk 像审讯清单 / Zack 问了他不知道的事 / 初见关系写错 / 指证像普通问答或推理 PPT" 与破梗同级阻塞。
AskUserQuestion：采纳全部→Phase 6；采纳部分→用户指定；保持原稿→结束。

### Phase 6 · 执行修改（可选）

按单元 spawn `dialogue-writer` 只改涉及部分；改后可选再跑 Phase 4（用户决定）。
