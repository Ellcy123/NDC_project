信息泄露检测

检查是否有不应在当前 Loop 出现的信息提前泄露给玩家。信息泄露直接违反"悬疑感优先"原则。

Loop 泄露内容 出处 应属 Loop 严重度
L2 emma_004 中 Zack 陈述"Morrison本人迷晕了你并拖入会客室"等完整结论 AVG/EPI01/Talk/loop2/emma_004.json 应由玩家从L1指证后自行整合 Critical
Critical L2 emma_004 Zack 主动陈述完整结论，替玩家做推理

在 emma_004 对话中，Zack 直接向 Emma 陈述"Morrison 本人迷晕了你并拖入会客室"等完整结论链。这些信息应该是玩家通过 L1 指证击破 Rosa 谎言后自行整合的推理成果，而非由 Zack 作为旁白直接宣读。

此问题直接违反设计原则一"悬疑感优先——绝不提前破梗"中"角色对话中不能替玩家说出结论"的核心要求。

修复建议

改写 emma_004，让 Zack 仅陈述已有证据事实（如"我在后台走廊发现了迷药毛巾和拖痕"），由 Emma 追问后逐步展开，而非一次性倾倒结论。玩家此时应已完成 L1，能自行理解对话暗示。

属性一致性检查

核对同一证据在不同数据源中的物理属性（日期、地点、描述等）是否前后一致。

证据ID 属性 Loop 1 描述 Loop 2 描述 结果

1115 地点 ItemStaticData: 后台走廊 loop1_rosa.json: 一楼走廊 Critical 矛盾
1114 → 1701 描述 沾湿的毛巾 → 有甜腻气味 一致 PASS

Critical 证据 1115 日期+地点 证据 1115 存在日期与地点双重矛盾

日期矛盾: ItemStaticData 记录为 11月15日，但 loop1_rosa.json 对话中引用为 11月4日。两处数据不一致将导致玩家在时间推理中得出错误结论。

地点矛盾: ItemStaticData 记录地点为"后台走廊"，而 loop1_rosa.json 对话中提及"一楼走廊"。走廊名称不统一会让玩家困惑这是同一地点还是两个不同位置。

修复建议

确认 1115 号证据的正确日期和地点后，统一 ItemStaticData 和所有引用该证据的对话文件。建议以 ItemStaticData（主数据源）为准，修改 loop1_rosa.json 中的引用。

单层真相原则

每个 Loop 只应揭示一层真相。验证实际揭示内容是否超出设计阈值。

Loop 应揭示的真相层 实际揭示 结果 问题
L1 Rosa 是目击者，被 Morrison 威胁沉默 Rosa 目击 Morrison 迷晕 Zack，出于恐惧沉默 PASS 符合单层设计
L2 Morrison 被收买，有预谋 Morrison 因赌债被 Whale 收买，预谋迷晕 Zack WARN 同时揭示"被收买"+"预谋"+"Whale 存在"+"Webb 约 Zack 真实目的"，信息量偏多
Minor L2 单层真相 L2 揭示信息量接近超标

L2 的核心真相层是"Morrison 被收买且有预谋"，但实际揭示内容涵盖了 4 个信息维度:

  1. Morrison 因赌债被收买（动机层）
  2. 迷晕 Zack 是预谋而非临时起意（性质层）
  3. Whale 的存在及其幕后角色（人物层）
  4. Webb 约 Zack 的真实目的（事件层）

虽然这 4 个维度在逻辑上紧密关联，构成一个完整的因果链，但从玩家认知负荷角度看处于边界水平。当前设计尚可接受，因为各维度之间有清晰的因果关系而非独立真相。

观察

如果后续 Loop (L3-L6) 也出现类似的多维度揭示，建议回头拆分 L2 的信息，将"Whale 的存在"延迟到 L3 揭示。但如果这是个案而非模式，当前设计可以保留。

Timeline Audit 总结

发现 2 个 Critical 级问题（证据 1115 日期/地点矛盾、emma_004 信息泄露）、1 个 WARN（rosa.md 措辞歧义）、1 个 Minor（L2 信息量偏多）。Critical 问题需要优先修复，直接影响玩家推理正确性和悬疑体验。
