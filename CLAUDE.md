# NDC_project

1920 年代芝加哥背景的侦探推理游戏 **内容设计与预配置工作区**。玩家扮演侦探 Zack Brennan，通过对话、证据收集、时间循环机制解谜。

本仓库产出（对话、证据、配置表等）最终同步到 `D:\NDC`（Unity 工程）。

## 核心数据架构

### 层级关系

```
Chapter (章节: EPI01/EPI02/EPI03)
 └── Unit (单元: Unit1/Unit2/Unit3) — 与 Chapter 一一对应
      └── Loop (循环: loop1-loop6) — 每章 6 个循环，逐步揭示真相
           └── Scene (场景) — 每个循环包含多个调查场景
                └── Talk / Expose (对话 / 指证) — 场景内的交互内容
```

### ID 编码规则

**证据/道具 ID（ItemStaticData）：**
- EPI01: 1xxx 系列（loop1=11xx, loop2=12xx, ...）
- EPI02: 2xxx 系列（loop1=21xx, loop2=22xx, ...）
- 派生证据: 17xx (EPI01), 27xx (EPI02)

**对话 ID：**
- 9 位数字，格式: `{loop}{scene}{sequence}`
- 例: 105001005 = loop1, scene 05(Tommy), 第 005 句

**证词 ID (TestimonyItem)：**
- 7 位数字，格式: `{loop}{npc}{sequence}`
- 例: 1031002 = loop1 的 03 号 NPC 的第 002 条证词

### 数据格式分工

| 格式 | 用途 |
|------|------|
| **YAML** | 循环配置 (loop1.yaml)、story/ 目录下的游戏数据、Preview 数据 |
| **JSON** | AVG 对话文件 (Talk/Expose)、preview_new2/data/table/ 下的预览数据 |
| **XLSX** | 策划填写的配置表原始格式，通过脚本转为 YAML/JSON |
| **MD** | 设计文档、对话草稿（Phase 1 工作格式） |

## 主要工作流

### 1. AVG 对话（Talk / Expose）

**严格遵守两阶段流程：**

- **Phase 1**: 只修改 MD 草稿（`AVG/对话配置工作及草稿/Loop{1-6}_对话草稿.md`），不碰 JSON
- **Phase 2**: 用户明确指示后才执行 `sync_to_json.py` 同步到 JSON

详细的 MD 格式规范和审查清单见 `.claude/commands/avg-dialogue.md`（slash command: `/avg-dialogue`）。

**修改对话前必须先查阅对应章节的设计文档：**
| 章节 | 设计文档 |
|------|---------|
| EPI01 | `第一章内容/` |
| EPI02 | `第二章内容/` |
| EPI03 | `第三章内容/Unit3/` |

绝不凭记忆编写剧情细节。

### 2. 证据与谜题设计

设计文档 → 配置表（XLSX）→ 脚本转换 → 预览验证 → 同步到 D:\NDC

关键约束：
- 每个场景/NPC 承载 1-3 个核心信息点，不超过 3 个
- 每轮循环只揭示一层真相，不在同一 Loop 解决多层疑问
- 疑点(Doubt)解锁需要至少两种不同来源的信息交叉验证

### 3. 配置表

- `story/` 目录维护 XLSX + YAML 双格式
- 用 `story/excel_yaml_converter.py` 进行 XLSX ↔ YAML 互转
- 用 `scripts/config_table_converter.py` 进行专项转换
- 最终通过 copy 命令同步到 `D:\NDC\Config\Datas\story\`

## 游戏设计三大原则

以下三条原则是本项目一切内容设计（剧情、对话、证据、谜题、指证）的最高优先级准则，高于所有具体规则。

### 原则一：悬疑感优先——绝不提前破梗

- **这是游戏，不是电视剧。** 玩家必须自己思考、自己发现、自己推理。
- 角色对话中不能替玩家说出结论。Zack可以出示证据、指出矛盾，但不能像旁白一样解释"所以这说明……"。玩家要自己把逻辑链补完。
- **信息揭示严格按循环节奏**：每个Loop只揭示一层真相。后续Loop才揭晓的信息，在当前Loop中必须完全隐藏或伪装成无关细节。
- **藏梗 > 明示**：能藏的线索就藏（如脚印深浅暗示两人体重不同），让玩家回头看时恍然大悟。指证击破谎言后玩家"大概率会知道"，但在击破之前绝不能让答案显而易见。
- 对话中NPC的谎言、退守、狡辩都必须是NPC主动说出的自然反应，不是Zack喂话后的被动否认。

### 原则二：信息严密性——零矛盾、零漏洞

- 所有证据的物理属性（尺寸、时间、距离、品牌、价格）必须前后一致，跨循环引用时不能出现数值矛盾。
- 证词原文与提取摘要信息密度对等，不丢失关键限定词（时间、地点、人物、条件）。
- 每个NPC的陈述在其知识范围内必须自洽——即使在说谎，谎言本身也要有内在逻辑。
- 时序严格：证据获取时间不能早于其存在时间；角色不能引用尚未发生的事件。

### 原则三：逻辑闭环——每个谜题必须有严密指向 + 多方暗示

- **单一证据不定案**：每个关键结论至少需要两种不同来源/类型的信息交叉验证（物证×证词、多方证词交叉、物证×物证等）。
- **多方暗示、多种渠道**：同一个真相可以通过证据、环境细节、NPC反应、物品描述等不同渠道给出暗示，但每个暗示单独看都不足以确定结论——只有组合后才指向唯一答案。
- **谎言措辞精确匹配证据维度**：每轮指证的谎言必须精确到只有对应维度的证据能反驳。同一维度的证据不跨轮拆分，每轮用不同维度的证据打不同论点。
- **陷阱有逻辑**：错误选项（陷阱证据）不是随机干扰项，而是有明确的"看起来相关但实际不构成反驳"的逻辑原因，让玩家事后能理解为什么选错了。

## 对话设计核心规则（17 条）

> 完整版含示例见 `AVG/对话配置工作及草稿/AVG对话配置规则.md`

### 一、分支对话规则（规则 1-5）

**1. 分支必须指向不同信息维度**

每组分支选项必须分别指向不同类别的信息点：

| 维度 | 说明 | 举例 |
|------|------|------|
| 身份/关系 | 人物背景、职业、与其他角色的关联 | "Webb是做什么生意的？" |
| 时间线 | 具体时间点、在场证明、行动轨迹 | "今晚你在哪儿？" |
| 事件/线索 | 发生的异常事件、可获取的物证 | "昨晚有什么异常？" |
| 情绪/态度 | 角色的心理状态、威胁暗示、动机 | "你在害怕什么？" |
| 背景/环境 | 人物来历、场所信息、时代背景 | "Foster什么来头？" |

**2. 分支内可独占获取关键信息**

不同分支路径允许有各自独占的 `get` 获取（证据/证词/时间线标记），玩家单次对话只能走一条路径。这是鼓励回访 NPC 的核心机制。

**3. Repeat 对话补完未走过的分支**

每个 NPC 的 `_repeat.json` 中的分支选项应指回首次对话中的分支路径，确保玩家不会永久错过任何信息。

**4. 语气差异影响演出但不影响核心信息**

当分支是"态度选择"时，两条路径最终获得的关键信息（keyInfoType/keyInfoContent）应当相同，但 NPC 反应、情绪表现、台词措辞可以不同。

**5. 所有分支最终汇合到同一节点**

禁止出现分支导致对话永久分叉或遗漏后续内容。

### 二、信息获取层级规则（规则 6-8）

**6. 证据有「直接获取」和「派生获取」两层**

| 层级 | 获取方式 | 举例 |
|------|---------|------|
| 直接获取 | 对话中 `script:"get"` 或场景物品点击 | 1114(沾湿毛巾), TIM001(枪声时间) |
| 分析派生 | 对直接证据执行"分析"操作 | 1114→1701(氯仿毛巾) |
| 合成派生 | 两个证据组合 | 1320+1425→1429(综合鉴定报告) |

**7. 每个场景/NPC 承载 1-3 个核心信息点**

| 信息点数 | 适用场景 |
|----------|---------|
| 1 | 过渡场景、补充信息场景 |
| 2 | 标准调查场景 |
| 3 | 核心剧情场景（首次对话、关键 NPC） |

**8. keyInfoType 标记必须准确分类**

每句包含关键信息的对话通过 `keyInfoType` 标记其信息类型：
- `timeline`：时间线相关（时间点、行动轨迹）
- `statement`：陈述类（声明、否认、自述）
- `identity`：身份/关系类（职业、人物关联）

### 三、跨循环信息递进规则（规则 9-12）

**9. 每轮循环只揭示一层真相**

| Loop | 揭示层 | 核心发现 |
|------|--------|---------|
| 1 | 犯罪事实确认 | 有人被下药、被拖拽、被枪杀 |
| 2 | 执法者腐败 | Morrison时间线矛盾、提前准备工具、赌债+Whale |
| 3 | 经济犯罪网络 | Webb的勒索帝国、假古董交易 |
| 4 | 嫌疑人浮现 | Jimmy偷盗、收Whale的钱、无不在场证明 |
| 5 | 情感动机补全 | Vivian被背叛、曲目掩护、复仇日记 |
| 6 | 真相拼图 | 弹道+音量+门锁+排班+Whale雇凶 |

**10. 疑点(Doubt)要求跨类型信息交叉验证**

解锁一个疑点至少需要两种不同来源/类型的信息组合：

| 交叉模式 | 举例 |
|----------|------|
| 证词 × 物证 | 时间线证词 + 物品 → 疑点 |
| 物证 × 物证 | 分析后毛巾 + 氯仿瓶 → 疑点 |
| 多方证词交叉 | 三人证词组合 → 疑点 |
| 证词 × 财务证据 | 关系网 + 额外收入 → 疑点 |

**11. 指证(Expose)遵循「递进击穿」结构**

每轮指证用不同维度的证据逐轮反驳同一条核心证词，从「事实矛盾 → 物证逻辑 → 物理排除」逐步深入。

**12. 后续 Loop 可复用前序 Loop 的证据**

跨 Loop 证据复用是核心机制，最终 Loop 应从几乎所有前序 Loop 调取证据形成完整拼图。

### 四、证词配置规则（规则 13-14）

**13. 提取的证词道具信息量必须与原文匹配**

- Testimony.words（证词原文）和 TestimonyItem.testimony（提取摘要）信息密度必须对等
- `<special>` 标记内容应与提取道具的核心陈述高度一致
- 提取道具可精简措辞，但不应丢失关键限定词（时间、地点、人物、条件）
- 提取道具不应包含原文中未明确表述的推断或结论

**14. 不要写「看起来能提取但实际不能提取」的信息**

- 凡包含具体时间点、地点、人名关联、明显矛盾的句子，要么配上对应 TestimonyItem 使其可提取，要么改写措辞使其不像线索
- `<special>` 标签标记的内容必须 100% 可提取
- 核心原则：**信息的"可提取感"和"实际可提取性"必须一致**

### 五、补充规范（规则 15-17）

**15. 证词、谎言与指证的关系**

```
Talk: 收集证词（get → TestimonyItem）
  → Zack 发现证词可疑 → 进入 Expose

Expose Round 1: Lie = Talk 中收集的证词（嫌疑人重申）
  → 玩家出示证据揭穿
  → 嫌疑人被迫承认部分事实，主动缩小范围（新谎言）

Expose Round 2: Lie = 嫌疑人的新狡辩（非预先收集）
  → 玩家出示新证据揭穿 → 依此递进
```

- 每个 Expose 的第一个 Lie 必须对应 Talk 中已收集的证词
- 后续 Lie 是嫌疑人在 Expose 中被逼出的新谎言
- **谎言由嫌疑人主动说出**（止损），不是 Zack 喂话后被动否认
- 谎言的揭穿必须由玩家通过出示证据完成，不能在 Talk 中由 Zack 直接戳穿

**16. 获取证据的节奏控制**

两次 `script:"get"` 之间至少隔 3-5 句普通对话，给玩家消化信息的空间。如果需要获取多个证据，应通过分支机制分散到不同路径。

**17. 视频资源路径一致性**

每句对话的视频字段必须与所属对话文件保持一致：
- `videoEpisode` = 章节标识（EPI01/EPI02）
- `videoLoop` = 循环标识（loop1-loop6）
- `videoScene` = 对话文件名（不含 .json）
- `videoId` = 对话 ID（与 `id` 字段一致）

- **NDC_project** = 内容设计、预配置、预览验证
- **D:\NDC** = Unity 游戏工程，包含运行时代码和最终资源
