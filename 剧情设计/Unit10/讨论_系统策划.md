## 四、三个机制特例的实现方案

### 4.1 L6 诱导招供系统

大纲描述：诱导槽 0–100%，3 个选择节点（+30/+40/+30），100% 触发 QTE。

#### 实现方案

**状态机设计**：
```
InduceState {
  value: int   // 0-100
  phase: 0/1/2/3/4  // 0=未激活, 1=节点1, 2=节点2, 3=节点3, 4=QTE触发
}
```

诱导槽是一段嵌入 Expose R3 内部的专属子流程，不是独立系统。建议在现有对话系统（Talk）的 script 字段扩展支持，或在 ExposeData 层新增字段。

**两种实现路径的权衡**：

路径 A（对话系统内实现）：在 script 类型里新增 type=9（诱导节点），ParameterInt 携带槽值增量（+30/+40/+30），对话链依赖 InduceState.value 做条件分支。
- 优点：不需要新配置表，沿用现有 Talk 链
- 缺点：script type 扩展需要程序改动；诱导槽 UI 层需新增

路径 B（ExposeData 扩展）：在 ExposeData 内新增 induce_rounds 字段，定义 3 个节点的 correct_talk_id / wrong_talk_id / value_delta。
- 优点：配置集中，逻辑清晰，不污染 Talk 表
- 缺点：ExposeData 结构新增字段，程序需配合解析

**建议路径 B**，新增字段示意：
```yaml
# ExposeData 新增字段（仅 R3 使用）
induce_system:
  enabled: true
  nodes:
    - id: 1
      correct_talk_id: 100XXXXXX   # 温和追问对话链
      wrong_talk_id:  100XXXXXX    # 强硬威胁对话链
      value_delta: 30
    - id: 2
      correct_talk_id: ...
      wrong_talk_id: ...
      value_delta: 40
    - id: 3
      correct_talk_id: ...
      wrong_talk_id: ...
      value_delta: 30
  qte_trigger_value: 100
  qte_talk_id: ...   # 100% 触发后的 QTE 演出对话链
  qte_duration_sec: 3
```

**UI 层**：诱导槽是专属进度条 UI，在 R3 阶段插入，R3 结束后隐藏。与现有 Expose 进度指示器并列显示。

**"喂话禁止"豁免**：仅在 induce_system.enabled=true 的节点内豁免（明确标注为"机制特例"）；L6 其他对话仍严格遵守 17 条规则。

**配置表新增需求**：
- ExposeData 新增 `induce_system` 可选字段（仅 L6 R3 使用，其他轮次为 null/false）
- Unity 端需对应新增 InduceSystemData 解析类

#### 已知风险
- QTE 3 秒按键在移动端（如无实体键盘）的适配方案未定义
- 错误选项（强硬威胁）是否清空/减少诱导槽？大纲未说明，建议：错误选项不增加槽值但不减少，以保证玩家最终能到达 100%（防止无法推进）

### 4.2 L5 律师/侦探二选一分支

大纲描述：选律师→三幕坏结局过场→回到选择点；选侦探→主线。

#### 实现方案

**分支节点**：在 ChapterConfig（loop5）的 exposes 之后，挂一个分支对话节点。使用现有 Talk 表的 script=1（分支选择）实现。

分支对话结构：
```
talk_id: 10500X001  (L5 Mickey 离场后 Zack 内心独白选择)
  script: 1         (分支选择)
  ParameterStr[0]: "接受 Mickey 的邀请，回法庭当律师"
  ParameterInt[0]: → 跳转到坏结局过场链 (bad_ending_talk_id)
  ParameterStr[1]: "留下来，继续当侦探"
  ParameterInt[1]: → 跳转到主线继续 (main_line_continue_id)
```

**坏结局过场**：3 幕黑屏白字 + 静态画面，用现有 Talk 链 script=8（切换场景）+ 专用 ending scene 实现。三幕连续播放后，触发 script=2（结束/回到选择点）——注意这里的"结束"是局部结束，回到选择 talk_id，不是 Unit 结束。

**"不锁存档"的实现**：
- 当前系统是否支持"分支结束后回到分支起点"需与程序确认。
- 建议：坏结局末尾的 Talk 节点 script 类型不设 2（结束游戏），而是跳回到选择 talk_id（ParameterInt 回指），形成可重选的软环。
- 存档点建议设在选择之前，不在坏结局三幕内部。

**配置表需求**：
- Talk 表新增坏结局三幕对话链（约 15–20 条 Talk 记录）
- ChapterConfig loop5 的 exposes 后追加分支钩子（或在 initTalk 里安排）
- 不需要新增配置表类型，现有 Talk 链可承载

### 4.3 L4 Lula 戏剧性登场

大纲描述：指证 Danny R3 谎言被击破→Lula 推门→签放弃继承协议→Danny 崩溃。

#### 实现方案

**触发点**：ExposeData R3 的 talkId（成功对话链）内，在击破 Danny 谎言后插入 interlude 事件。

具体链路：
```
ExposeData (Danny, R3)
  → talkId → 击破对话链
  → 对话链末尾 script=8 (切换场景: 触发 Lula 登场演出)
  → interlude talk: 门外脚步声 → Lula 推门 → 签字特写 → Danny 崩溃
  → 演出结束后 script=8 切回赌场场景，进入 post_expose 剧情
```

**"Mickey 提前安排 Lula 候命"的系统含义**：Lula 在 L4 开篇已经在 Margaret 鞋坊（场景 1001）出场，玩家可以提前与她对话，但她不知道自己要被"安排"去赌场——这个安排是 Mickey 的场外行动，属于剧情设定，不需要系统层面预埋可见触发。实现上只需保证 Lula 在 R3 指证后的对话链能被找到（Talk 中 Lula 的 NPC ID 已注册在 NPCStaticData 中）。

**签字协议**：不需要新物品（签字动作是演出，非可收集证据）。但可以考虑在签字后派生一条证词 TestimonyItem（Lula 当场放弃继承——作为 Danny 无动机论的补充证言），有助于补全证据链。建议派生 TestimonyItem 而非 ItemStaticData 物品。

**配置表需求**：仅需 Talk 链配置（R3 成功后 interlude），无新表。

## 五、5 个小玩法的实现方案

### 5.1 L2 连号钞票拼图

**机制描述**：玩家获得两组钞票（O'Hara 的 A73625900–904B / 910–914B，缺 905–909B；Vinnie 的 905/906/908B），在 UI 中拖拽或点击对比编号，发现 Vinnie 的号码填补缺口。

**输入**：ItemStaticData 10202（O'Hara 贷款钞票）+ ItemStaticData 10206（Vinnie 的钞票）

**输出**：触发一次对话/提示节点（Talk script=3 type，获取派生 TestimonyItem 或直接解锁疑点）。不产出新 ItemStaticData；推理结论通过 TestimonyItem 形式体现（"连号钞票拼合分析"可作为 10201 的派生 TestimonyItem）。

**UI 需求**：编号对比表 UI + 缺号高亮 + 填补动画。这是专属小玩法 UI，需要程序新增。

**配置表新增**：
- 建议在 ItemStaticData 中为 10202 设置 canCombined=true，combineParameter=[10202, 10206]，合成产出派生 ID（107xx 段）
- 或：不走合成系统，改为 minigame 触发器，由程序处理 UI 逻辑，完成后自动给予派生 TestimonyItem
- 建议选后者（minigame 模式），更符合"拼图"的玩法感，不强行纳入 combine 系统

**是否产出派生证据**：产出 TestimonyItem（连号钞票拼合分析），不产出新 ItemStaticData 物品。

### 5.2 L2 赌博小玩法

**机制描述**：跨章可玩的赌博小游戏（禁酒令时期常见玩法），Tony 作为信息源偶尔给出提示。

**输入**：无固定证据输入。玩家在 Silver Moon 赌场（1015）与 Tony 对话选择进入。

**输出**：
- 额外 Tony 证词（非主线必须，但补充背景信息）
- 跨章节：Unit10 后仍可重入赌博小玩法（不影响主线进度）

**配置表新增**：
- 需要 Talk 表增加进入赌博小玩法的入口节点（script=8 切换到小玩法场景）
- 赌博小玩法本身是独立 minigame，不属于当前 16 张配置表范畴；需与程序确认是否已有复用框架（Unit2 是否已有）
- Tony 的 LoopTalkInfo（重复对话）应包含"进入赌博"选项，跨章节持久化

**风险**：L2 赌博小玩法与"Unit10 之后仍可玩"的持久化设计，需要 ChapterConfig 层面的跨章节解锁逻辑支持。当前系统未见跨章节小玩法持久化的先例，属于新功能需求，技术风险中等。

### 5.3 L3 笔迹对比 + 贷款计算器

**机制描述**：
- 笔迹对比：将 Frank 真迹（10301 牛奶订购签收单）与贷款档案签名（10307 LAKESHORE TRUST BANK 客户贷款档案）进行视觉对比，识别代签。
- 贷款计算器：将贷款协议第 5/7/12 条（10302/10303/10304）的数值代入，计算年化利率 → 产出派生分析报告。

## 七、L1 Foster 法医报告时序问题处理建议

**问题描述**：L1 Morrison 在指证 R3 声称"我有尸检调查信息可以证明死者身份"，但 Emma 转 Foster 的初步意见是"尸检最快明天才能出，连性别都未定"。Morrison 的话是虚张声势，这本身就是 R3 要击穿的谎言。

**时序自洽分析**：

大纲对此的处理实际上是自洽的——
- L1 Morrison：声称有尸检信息 → 这就是谎言，R3 用 Foster 转述"报告未出"来击穿
- L5 末 Foster 告诉 Zack：法医鉴定结果已出，非火灾致死
- L6 Foster 在医院出场：提供法医鉴定报告（10601）实物

时序链：L1 案发当夜（~21:00）Morrison 胡扯 → L5（11/7 21:00）Foster 完成鉴定口头告知 → L6（11/8 09:00）Zack 取到报告实物

这条链是合理的：L1 到 L5 约 24 小时，基本符合"最快明天"的描述。

**需要处理的配置层问题**：

R3 击穿证据是 TestimonyItem（Emma 转 Foster 初步分析），不是 ItemStaticData 物品。该 TestimonyItem 的 triggerType 建议设为 0（直接获取），在 L1 电话亭场景 Emma 对话后直接给予，无需验证条件。

法医鉴定报告（10601）在 L6 才出现。系统层面需要确认：如果玩家在 L1 已经记住"Foster 说报告明天才能出"，L5 末 Foster 电话说"鉴定出来了"是否会感觉前后矛盾？

处理建议：
1. Emma 在 L1 的转述用词要精确："Foster 说至少明天才有初步结果，现在连性别都没法确认。" 这既可以击穿 Morrison（他不可能有报告），也为 L5 末"24 小时后 Foster 拿出结果"铺垫合理时间窗口。
2. L5 末 Foster 电话台词需强调"花了整整一天"，强化时间感，避免玩家觉得"怎么这么快"。
3. 10601 法医鉴定报告的 note 字段写明：`关键——L5 剧情揭示结论 + L6 指证 Morrison 兑现承诺时出示实物`，区分"结论已知（L5 口头）"与"报告实物（L6 取得）"两个时间节点。

**Morrison 在 L1 声称"有尸检调查信息"的谎言**：这是 Morrison 惯用的糊弄手段——他实际上什么都没有，只是用专业术语压制 Zack。R3 击穿逻辑自洽，无需额外处理。
