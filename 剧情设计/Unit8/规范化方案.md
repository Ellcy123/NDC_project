# Unit8 State 文件体系规范化方案

> 目的：将 `Unit8重构版0417/state/loop{1-6}_state.yaml` 里自造的场景 ID、疑点 ID、疑点 condition 格式规范化为游戏官方 DoubtConfig/SceneConfig 规范。
>
> 参考规范：`docs/游戏系统/核心玩法/疑点系统.md`、`docs/枚举速查表.md`、`剧情设计/Unit3/state/loop1_state.yaml`
>
> 本方案为**规划文档**——不修改任何 state 文件。执行落地应在用户确认后另行处理。

---

## 0. 规范速览

| 项 | 旧格式（Unit8 当前） | 新格式（官方规范） |
|---|---|---|
| 场景 ID | `s101 / s201 / s301 / ...`（字符串） | 整数 `{unit}0{seq 01-99}` — Unit8 用 `10xx` 段 |
| 疑点 ID | `d1001–d1011`（字符串） | 整数 `{unit}{loop}{seq}` — 如 `1101` |
| 物品 ID | `1101–1604`（整数） | ✅ 已规范，保持不变 |
| 疑点 condition | 自由文本 `unlock_condition: "需要两种..."` | 简写字符串：`"item:xxxx + testimony:xxxxxxx + relation:xxx"`（符合 Unit3 state 格式；state_to_preview 期待将其展开为 `[{type, param}]` 数组） |

**条件类型枚举（DoubtConditionType）**：
- `item:{物品ID}` — type=1（Item，ItemStaticData.id）
- `relation:{NPCID1-NPCID2}` — type=2（RelationNetwork，TestimonyItem.id 实际走证词—拖到 NPC 头像；简写里用 NPC 对表达"需要建立该关系"）
- `testimony:{证词ID}` — type=3（Timeline，TestimonyItem.id，需要在时间线上标记）

### Loop5 灵魂黑夜三疑点（1501/1502/1503）的 Condition 可行性分析

**关键决策：大纲设计是"loop5 开篇一次性 UI 浮现"，但官方机制要求 condition 触发**——本方案选择以 loop5 **进入开场时** 玩家已经累积的证据作为"默认满足"的 condition。

玩家进入 loop5 opening（灵魂黑夜）时已持有：
- 全部 loop3 证据（威胁纸条 1306 含字迹 J、女儿医疗单 1307）
- 全部 loop4 证据（被撕毁合同 1401 字迹 J，Vivian 的 4011004/4011005/4011006 三条后续证词，弹壳 1308 已被 Morrison 夺走但鉴定信息已获得）

**当 condition 满足时，疑点自动解锁并弹 UI**——若玩家 loop3/loop4 正常推进，这三条 condition 在进入 loop5 时全部满足，表面上就是"开篇一次性浮现"。即：

- 1501（恐吓 Rosa 的人是谁）：`item:1306 + item:1401` — 进入 L5 时玩家手中两件物证都有
- 1502（勒索 Whale 的执行人是谁）：`item:1401 + testimony:5061001` — 注意 5061001 是 L5 Anna 证词，所以严格意义上这个疑点 L5 开篇还差一个 condition，解锁时机为 L5 玩家到 Jimmy 家采访 Anna 之后。大纲"开篇浮现"效果需要技术变通（见风险点 R2）
- 1503（凶手如何杀 Webb）：`item:1503 + testimony:4011006` — 类似问题，1503 鼓点单子是 L5 Tommy 办公室取得的，开篇也还差 condition

**变通方案（见风险点 R2）**：
- 方案 A（推荐）：L5 opening UI 浮现时这三个疑点设为"visible=true, unlocked=false"，即玩家可见但 EXPOSE 功能未启用。玩家自由探索后各自满足 condition 再解锁。与游戏的"当前目标疑点 vs 其他疑点"机制自然兼容。
- 方案 B：调整 condition 到 L4 末能满足的证据上（如 1502 用 `item:1401 + testimony:4011004` 即 Vivian 的"帮 Whale 勒索的不是我"证言），使三疑点 L5 开篇就全部解锁——但这牺牲了 Anna 证词这个关键证据的戏剧价值。

建议：**方案 A**——保留 Anna/Tommy/Vivian 各自证据的戏剧权重，L5 opening UI 只做可见化浮现，condition 满足在自由探索中完成。

```yaml
- id: 1101
  text: "Rosa 声称目击..."

  status: "unlocked_after_expose_round1"
```

### 4.5 关联文本中的 `d1xxx` 引用（需要替换为新 4 位 ID）

以下位置含对疑点的文本引用，需同步替换：

- `player_context.post_expose_knowledge`：`"d1003 解锁：..."` → `"1201 解锁：..."`（6 个 loop 都有）
- `player_context.new_directions.trigger`：如 loop4 中 `"疑点 d1002 / d1006 驱动"` → `"疑点 1102 / 1401 驱动"`
- `doubts[*].resolution`：如 `"d1011 解锁..."` 引用
- `doubts[*].unlock_condition`（自由文本转简写时，引用 d1xxx 的地方一并替换）
- `npcs.*.player_inquiry.*`：含 `"（d1004 驱动）"` / `"（疑点 d1002 驱动）"` 之类标注——**共计约 15 处**，需全部替换
- `connoisseur.note`：含"考 d1xxx 解锁路径"之类文字（偶有）

### 4.6 `lie_source` / `player_inquiry` 中的场景引用（较少）

`lie_source` 引用的是证词 ID 或自由文本说明，不涉及场景 ID。唯一需要关注的是 `pre_turn_trigger` 这类自造字段里如果提到 `s405` 要改。

### 4.7 `doubts` 段不统一问题的修正

当前 loop2 的 d1003 用的是 `posed_condition` 字段（不是 `unlock_condition`），loop3 的 d1003 用的是 `posed_loop` + `resolution_condition`——这些是大纲追加的非规范字段。规范化时：
- 统一字段名为 `unlock_condition`（表达"如何让这个疑点 unlock"）
- 增加可选字段 `posed_loop` / `resolved_loop` 作为 meta 信息（不影响游戏数据，只影响 state_to_preview 展示）

## 5. 风险点与决策点

### R1. Loop5 灵魂黑夜三疑点"开篇浮现"与 condition 机制的冲突

**问题**：大纲将 d1007/d1008/d1009 设计为"loop5 opening 一次性 UI 浮现"，但官方 DoubtConfig 要求 condition 触发解锁。

**建议**（见第 3 节末尾）：采用方案 A——loop5 opening 时这三个疑点只显示 UI 预览（visible=true，unlocked=false），condition 满足在自由探索过程中完成。这一设计与游戏的"当前目标疑点 vs 其他疑点"机制兼容，也保留了 Anna/Tommy/Vivian 证据各自的戏剧权重。

**等价的技术实现**：
- state 里 1501/1502/1503 的 `status` 设为 `"visible_in_loop5_opening_unlock_on_evidence"`
- 游戏代码层面 loop5 opening 触发 UI 动画展示四个疑点图标；真正 unlock 判定走 condition。

**需用户决策**：确认采用方案 A，还是方案 B（调整 condition 到 loop4 末已有的证据）。

### R2. 1505 英语练习书是 envir 能否用作 condition？

**问题**：1505 是 `type: envir`, `pickup: false`——玩家无法"拥有"它，只是 Zack 观察记忆。按 DoubtConfig 的 `type=1 Item` 定义"需拥有 ItemStaticData 中的指定物品"，envir 道具理论上无法作为 `item:1505` condition。

**当前方案已回避**：1501 的 condition 用 `item:1306 + item:1401`（两件可拾取物证的字迹已经足够解锁），不依赖 1505。1505 的字迹闭环作为 post_expose_dialogue 的叙事高潮保留。

**需用户决策**：确认是否这样处理，还是把 1505 的 type 改为 `item + pickup:true`（但这与大纲的"Zack 凭观察记忆陈述，不作为物证出示"定性矛盾）。

### R3. Loop3 一楼会客室（1001）场景 type 从 limited_exploration 变 free_exploration

**问题**：L1 s101 是 `limited_exploration`（Morrison 管控现场），L3 s303 是 `free_exploration`（Morrison 勘验后开放）。同一物理 ID 1001 在不同 loop 有不同 type。

**处理方案**：state_to_preview 脚本读取的是"每个 loop 内 scenes.[N].type"——允许同一 ID 在不同 loop 呈现不同 type，这是合理设计（物理空间同，游戏状态不同）。SceneConfig 配置表可按"Loop-Scene 组合"建立条目，不必强行统一。

**不需用户决策**——保留现状即可。

### R4. 酒吧大厅（1007）与一楼走廊（1009）是同一场景还是独立场景？

**问题**：s302（loop3 大厅，Morrison 勘验汇报）、s401（loop4 一楼走廊，垃圾桶附近发现合同）、s405（loop4 大厅等待区 + turn_cutscene）在大纲里并未明确说明是否同一物理空间。

**初步方案**：按语义区分：
- **1007 酒吧大厅** = 公共聚会空间（舞池/大堂），Morrison 在此等待或勘验汇报（s302/s405）
- **1009 一楼走廊** = 通道（包括垃圾桶所在的一楼侧廊），1401 合同发现点（s401）
- L5 s502（厨房）也在一楼，但物理独立，归 1003 不冲突

**需用户决策**：1007/1009 是否合并？如果用户认为"一楼走廊就是大厅的一部分"，则 s302/s401/s405 全部归 1007。目前按独立处理，更贴近大纲描述（垃圾桶通常在通道而非大厅）。

### R5. `first_scene: suspect_suicide_sequence` / `first_scene: ending_sequence_act3` 这类自造值

**问题**：`first_scene` 字段预期是场景 ID（整数），但当前 state 里 1507 写的是 `first_scene: suspect_suicide_sequence`，1604 写的是 `first_scene: ending_sequence_act3`——这是自造名称，不是场景 ID。

**处理方案**：
- 1507 的物理获得地点是"蓝月亮酒吧后门"（Jimmy 自杀现场）→ `first_scene: 1011`
- 1604 的物理获得地点是"蓝月亮酒吧会客室保险箱"→ `first_scene: 1001`
- 保留 `obtained_via` 或新增 `trigger_event` meta 字段标注具体的剧情节点（"suspect_suicide_sequence" / "ending_sequence_act3"）

**不需用户决策**——这是技术处理，仅提示后续执行时要注意的细节。
