# Unit3 场景拆分规则

本文档记录 Unit3 场景文档的拆分口径，用于后续美术需求、AVG 预览网页和配置表落地时统一理解。

核心判断不是“这个地点出现在哪个 Loop”，而是“美术产出的资源形态是什么”。

---

## 一句话规则

- 探索场景：非 AVG 的物理地点底图类场景，按地点资源建文件。
- AVG 场景：给剧情对白 / 过场使用的完整画面，人物和场景是一体的，按 Loop + 剧情位置 + 剧情用途建文件。
- 突发事件：不是场景文件，保留在美术需求文档里，挂到某个场景或剧情节点上。
- 指证背景：优先复用探索场景或 AVG 资产，不默认单独开“指证背景”目录。

---

## 探索场景

探索场景放在 `探索场景/` 下，命名格式：

```text
EXP_SC{sceneId}_{地点名}.md
```

这里的“探索场景”是广义说法，包含自由探索场景、过渡走廊、可复用的指证场地。判断重点不是玩家能不能自由搜证，而是它的美术产出不是整张 AVG 图。

探索场景的美术产出通常是背景 / 环境底图，NPC、证据、可点击物件是分开的资源和配置挂载。

同一个探索地点即使跨多个 Loop 使用，也不因为 Loop 不同就拆成多个文件。Loop 差异写在同一个文件内部，比如 L3 有哪些证据、L4 新增什么小玩法、L6 是否新增某个 NPC 事件。

如果两个“地点名”从美术产出看实际是同一张底图 / 同一处入口，只因为剧情阶段不同才被拆开，应优先合并为一个探索场景。Unit3 中 SC3013 公寓楼门口已并入 SC3001 Thomas 公寓楼下入口；L2 Mary 拦截、搜身观察和指证挂载在 SC3001 / 3201，不再单独出 SC3013 底图。

只有当美术产出真的变成另一套资源时，才考虑拆出新文件或资源变体。Unit3 探索底图的时间只分两档：白天 / 夜晚。清晨、上午、中午、下午统一归白天；傍晚后、案发回溯、夜间调查统一归夜晚。

日夜变体不改变场景文档文件名，也不改变内部英文名。内部英文名保持短名，例如 `u3_exp_rooftop`；具体背景资源名在配置里加 `_day` / `_night`，例如 `SC3002_bg_u3_exp_rooftop_day`、`SC3002_bg_u3_exp_rooftop_night`。

公寓楼动线相关地点必须写清白天 / 夜晚需求：SC3001 楼下入口、SC3016 一楼走廊、SC3006 二楼走廊、SC3015 三楼走廊、SC3002 楼顶。Helen 家因 L3/L4 指证跨日夜，也需要写白天 / 夜晚灯光差异。其他室内地点按实际开放搜证时间选择一版即可，不为了“每个地点都有两版”而硬拆。

## AVG 场景

AVG 场景放在 `AVG场景/` 下，命名格式：

```text
AVG_L{loop}_{剧情位置}_{地点名}_{剧情用途}.md
```

AVG 场景面向的是“剧情镜头”。人物、站位、表情状态、场景构图通常画在同一张完整图里，不按“空背景 + NPC 立绘 + 证据点”拆。

所以同一个地点可以有多个 AVG 文件。比如 Zack 侦探事务所在 L1 是 Mickey 正式委托，L6 可能是 Emma 回报社、Mary/Helen 道谢、Leonard 送文件；这些在美术上是不同的人物组合和戏剧关系，应当分开。

AVG 文件名里写 Loop，是为了让人一眼知道它服务哪一段剧情，不是要求所有探索场景也跟着按 Loop 拆。

AVG 文件名里不放 SC 编号。SC 编号、拟 AVG SceneId、ChapterConfig / ArtAssetConfig 用途写在文件正文里，避免把 AVG 误读成某个探索场景的 Loop 版本。

如果某段指证后内容在游戏内默认是黑屏对白、两人聊天或普通 NPC 对话，不因为剧情重要就建 AVG 文件。只有确认需要独立画面产出时，才进入 `AVG场景/`。

---

## 突发事件

突发事件动态漫画不是场景，仍放在：

```text
剧情设计/Unit3/美术需求/Unit3_AVG突发事件动态漫画.md
```

场景文件里可以写“这里挂载某个突发事件”，但不把突发事件拆成新的场景文件。

当前 Unit3 的典型挂载方式：

- L3 Helen 家：查看 Helen 口红后触发“Emma 点破 Helen 是出轨对象”。
- L4 Helen 家：补充指证后触发“Mary 登场为 Helen 求情”。
- L5 湖滨信托银行大厅：用独立纯 AVG 场景表现“Charles 带走 Emma”，随后 Zack 独自进入 Bernard 办公室；不再挂为办公室突发事件。
- L6 二楼走廊：经过时触发“Seamus 使用润滑油”。

## 指证背景

指证背景是配置挂载，不是默认独立场景分类。

大多数情况下，指证背景可以复用对应的探索地点或 AVG 资产：

- L1 Morrison 指证：复用警局相关画面。
- L2 Mary 指证：复用 SC3001 Thomas 公寓楼下入口相关画面；SC3013 公寓楼门口已并入，不再单独配置 / 出图。
- L3 / L4 Helen 指证：复用 Helen 家相关画面。
- L5 Bernard 指证：复用 Bernard 办公室相关画面。
- L6 Mickey 控辩对决：复用 Mickey 办公室相关画面。

只有当指证画面在构图、人物状态、镜头表现上明显不同，才作为单独 AVG / topBg 资产需求写清楚。

---

## 内部英文名

中文文件名给策划、美术和剧情阅读；每个场景文件内部必须写 `内部英文名`，给配置、资源命名和预览网页定位使用。

内部英文名使用短 snake_case：

- 探索场景：`u3_exp_{short_place}`，例如 `u3_exp_rooftop`。日夜差异体现在背景资源名，不塞进内部英文名。
- AVG 场景：`u3_l{loop}_{position}_{short_place}_{short_use}`，例如 `u3_l4_bridge_cityhall`。

英文名不追求完整翻译剧情句子，只要求短、稳定、可读。SC 编号和拟 SceneId 仍放在正文其他字段里，不塞进内部英文名。

---

## 和 state 的关系

state 继续保持给策划和剧情阅读的结构，不反向塞大量配置 ID。

场景拆分、AVG 资产命名、突发事件挂载、topBg 复用关系，先在场景 / 美术需求文档里说清楚。后续落配置时，再映射到 SceneConfig、ArtAssetConfig、ChapterConfig、Talk / Expose 入口。

这样 state 仍然负责“剧情为什么这样走”，场景文档负责“这段需要什么资源形态”，配置表负责“游戏和预览网页怎么挂载”。
