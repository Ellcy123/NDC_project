# NDC 第一章 配置表详解

> 本文档基于第一章（Unit1 / EPI01）的完整配置数据，详细说明 `Assets/table/` 下 **16 张活跃数据表** 的字段含义、ID 编码规则、表间关联关系，以及 Unity C# 代码的加载调用逻辑。

---

## 一、总览

### 1.1 表格清单

### 1.2 数据流

```
策划导出 JSON (Assets/table/*.json)
        ↓ Unity Editor 序列化
二进制文件 (Resources/table/*.bytes)
        ↓ TableConfig.LoadTableAsync()
运行时内存 (静态数组 C_xxx[] + 字典 Dic_xxx)
        ↓ Manager 层按需查询
游戏逻辑使用
```

### 1.3 加载架构

核心类：`Assets/Scripts/table/TableConfig.cs`

- **静态数组** `C_ItemStaticData[]`：保留原始顺序，支持按下标访问
- **静态字典** `Dic_ItemStaticData`：`Dictionary<int, IConfigTable>`，支持按主键 O(1) 查找
- **加载入口**：`LoadTableAsync()`（协程异步）或 `LoadTable()`（同步）
- **查询API**：
  - `GetTableByKey<T>(TableEnum, int key)` — 按主键查找
  - `TryGetTableByKey<T>(TableEnum, int key, out T)` — 安全查找
  - `GetTable<T>(TableEnum, int index)` — 按数组下标
  - `GetTabelData<T>(TableEnum)` — 获取整张表数组
  - `GetTables<T>(TableEnum, Func<IConfigTable, bool>)` — 条件过滤

## 二、ID 编码规则

### 2.1 场景ID（SceneConfig.sceneId）

**格式**：4位数 `{unit}{loop}{location}`

| 位 | 含义 | 范围 |
|----|------|------|
| 第1位 | 章节(Unit) | 1=第一章, 2=第二章 |
| 第2位 | 循环(Loop) | 1-6 |
| 第3-4位 | 地点编号 | 01-17+ |

**示例**：

| sceneId | 含义 |
|---------|------|
| 1101 | Unit1 Loop1 Rosa储藏室 |
| 1103 | Unit1 Loop1 Tommy办公室 |
| 1104 | Unit1 Loop1 Webb会客室 |
| 1117 | Unit1 Loop1 特殊场景（开场/过场） |
| 1201 | Unit1 Loop2 Rosa储藏室（同地点不同轮次） |
| 1210 | Unit1 Loop2 Morrison太太家 |

### 2.2 物品ID（ItemStaticData.id）

**格式**：4位数

| 位 | 含义 |
|----|------|
| 第1位 | 章节(Unit) |
| 第2位 | 循环(Loop) 或分类标记 |
| 第3-4位 | 物品编号 |

**特殊段**：
- `17xx`：分析后的证据（如 1701=分析后的沾湿毛巾）
- `18xx`：道具类（门等交互物品）
- `110xxx`（6位）：纯装饰/环境物品

### 2.3 NPC ID（NPCStaticData.id）

**格式**：3位数 `1XX`（第一章）

| id | 角色 | role值 |
|----|------|--------|
| 101 | Zack Brennan（主角侦探） | 4 (protagonist) |
| 102 | Emma O'Malley（搭档） | 4 (protagonist) |
| 103 | Rosa Martinez（嫌疑人/清洁工） | 2 (suspect) |
| 104 | Morrison（嫌疑人/警探） | 2 (suspect) |
| 105 | Tommy（嫌疑人/酒吧经理） | 2 (suspect) |
| 106 | Vivian Rose（嫌疑人/歌手） | 2 (suspect) |
| 107 | Jimmy（凶手/厨师） | 2 (suspect) |
| 108 | Anna（证人） | 3 (others) |
| 109 | Webb（受害者） | 1 (deceased) |
| 110 | Mrs. Morrison（证人） | 3 (others) |
| 111 | Whale（幕后主使） | 3 (others) |

### 2.4 NPCLoopData ID

**格式**：4位数 `{NPC序号}{场景实例序号}`

- 1001 = Tommy 在 Loop1 的实例
- 1002 = Rosa 在 Loop1 的实例
- 依次类推，每个 NPC 在不同场景/轮次中有不同的实例ID

### 2.5 对话ID（Talk.id）

**格式**：6-9位数

- **普通对话**：`{NPC3位}{对话组3位}{句序3位}`，如 `105001001` = Tommy 第1组对话 第1句
- **指证对话**：`{轮次2位}{序号4位}`，如 `110001` = Loop1指证 第1句
- **`next` 字段**指向下一句对话ID，形成链式结构

### 2.6 证词/证词道具ID

**Testimony.id** = 与 Talk.id 类似格式（嵌入在证词段落中）

**TestimonyItem.id** = 7位数 `{NPC3位}{轮次1位}{序号3位}`，如 `1031002` = Rosa Loop1 第2条证词道具

### 2.7 疑点ID（DoubtConfig.id）

**格式**：4位数 `{unit}{loop}{序号}`，如 1101=Unit1 Loop1 第1个疑点

### 2.8 ChapterConfig.id

**格式**：3位数 `{unit}{loop}{序号}`，如 101=Unit1 Loop1

## 三、16 张活跃表逐一详解

---

### 3.1 ChapterConfig — 章节循环主配置表

**用途**：每个循环(Loop)的顶层配置。定义该轮的初始对话、初始场景、疑点列表和指证方案。

**主键**：`id`

**C# 类**：`Assets/Scripts/table/ChapterConfig.cs`

| 字段 | C#类型 | JSON类型 | 含义 | 示例值 |
|------|--------|----------|------|--------|
| id | int | string | 章节循环索引 | `"101"` (Unit1 Loop1) |
| initTalk | int | string | 该轮开场对话ID → Talk表 | `"102001001"` |
| initScene | int | string | 该轮初始场景ID → SceneConfig表 | `"1117"` |
| doubts | DoubtConfig[] | object[] | 疑点列表（内联或ID引用） → DoubtConfig表 | 见下方 |
| topBg | string | string | 指证场景上方背景图路径 | `"Art/Scene/Expose/epi01/top_rosa"` |
| bottomBg | string | string | 指证场景下方背景图路径 | `"Art/Scene/Expose/epi01/rosa"` |
| exposeNpcId | int | string | 本轮指证目标NPC ID → NPCStaticData表 | `"103"` (Rosa) |
| exposes | ExposeData[] | object[] | 指证方案列表（内联或ID引用） → ExposeData表 | 见下方 |

**外键关联**：
- `initTalk` → **Talk** 表
- `initScene` → **SceneConfig** 表
- `doubts[]` → **DoubtConfig** 表（C# 中通过 `_doubtsIds` 懒加载）
- `exposeNpcId` → **NPCStaticData** 表
- `exposes[]` → **ExposeData** 表（C# 中通过 `_exposesIds` 懒加载）

**JSON 示例**（Loop1 — 指证Rosa）：
```json
{
  "id": "101",
  "initTalk": "102001001",
  "initScene": "1117",
  "doubts": [
    { "id": "1101", "condition": [{"type":"3","param":"1031002"},{"type":"1","param":"1115"}], "text": "Rosa当晚究竟在哪工作？" },
    { "id": "1102", "condition": [{"type":"1","param":"1701"},{"type":"1","param":"1121"}], "text": "这条毛巾为什么有奇怪的味道？" }
  ],
  "topBg": "Art/Scene/Expose/epi01/top_rosa",
  "bottomBg": "Art/Scene/Expose/epi01/rosa",
  "exposeNpcId": "103",
  "exposes": [
    { "id": "1", "testimony": "1031002", "item": ["1115"], "talkId": "110001" },
    { "id": "2", "testimony": "1031002", "item": ["1701","1121"], "talkId": "110003" }
  ]
}
```

### 3.2 SceneConfig — 场景配置表

**用途**：定义每个场景包含的NPC和道具，并通过 `location` 外键引用 **LocationConfig** 获取背景图、场景名称等地点信息。是数据的**"超级容器"**——NPC和道具仅通过此表与场景关联。

**主键**：`sceneId`

**C# 类**：`Assets/Scripts/table/SceneConfig.cs`

| 字段 | C#类型 | JSON类型 | 含义 | 示例值 |
|------|--------|----------|------|--------|
| sceneId | int | string | 场景ID（4位编码） | `"1103"` |
| location | LocationConfig | object | 地点信息 → LocationConfig表（含名称、背景图、音效） | `{"id":"103",...}` |
| backgroundMusic | string | string | 背景音乐资源路径 | `"test"` |
| unlockCondition | string | string | 解锁条件 | `""` |
| NPCInfos | NPCLoopData[] | object[] | 场景中的NPC实例 → NPCLoopData表 | 见下方 |
| ItemIDs | int[] | string[] | 场景中的道具ID列表 → ItemStaticData表 | `["110021","1111","1112"]` |
| note | string | string | 备注 | `""` |

**外键关联**：
- `location` → **LocationConfig** 表（C# 中通过 `_locationId` 懒加载）
- `NPCInfos[]` → **NPCLoopData** 表（C# 中通过 `_NPCInfosIds` 懒加载）
- `ItemIDs[]` → **ItemStaticData** 表（直接存储ID数组）

**被引用**：
- **ChapterConfig.initScene** 引用此表

**配置注意事项**：
- 场景名称、背景图、场景类型、环境音效均从 **LocationConfig** 获取，SceneConfig 本身不存储这些字段
- 同一物理地点在不同 Loop 中是不同的 sceneId（如 1101 和 1201 都是 Rosa储藏室），但引用同一个 LocationConfig
- NPCInfos 可选，不是所有场景都有NPC
- ItemIDs 中可能包含6位数的装饰物品ID（如 `"110021"`）

**JSON 示例**：
```json
{
  "sceneId": "1103",
  "location": {
    "id": "103",
    "Name": ["Tommy办公室","TommyOffice"],
    "sceneType": "1",
    "backgroundImage": "Art\\Scene\\Backgrounds\\EPI01\\SC003_bg_TommyOffice"
  },
  "backgroundMusic": "test",
  "NPCInfos": [{ "id": "1001", "NPC": {...}, "TalkInfo": {...}, "LoopTalkInfo": {...}, ... }],
  "ItemIDs": ["1131","1132","1133"]
}
```

### 3.3 ItemStaticData — 物品/证据静态数据表

**用途**：所有物品（证据、环境装饰、可交互道具、门等）的静态定义。包含名称、描述、位置、美术资源路径、分析/合成参数等。

**主键**：`id`

**C# 类**：`Assets/Scripts/table/ItemStaticData.cs`

| 字段 | C#类型 | JSON类型 | 含义 | 示例值 |
|------|--------|----------|------|--------|
| id | int | string | 物品ID | `"1114"` |
| Name | string[] | string[] | [中文名, 英文名] | `["沾湿的毛巾","Pungent Towel"]` |
| itemType | itemType(enum) | string | 物品类型枚举 | `"3"` (item) |
| ActionParam | string | string | 动作参数 | `""` |
| canAnalyzed | bool | string | 是否可分析 | `"true"` |
| analysedEvidence | int | string | 分析后变为的物品ID → ItemStaticData | `"1701"` |
| beforeAnalysedEvidence | int | string | 分析前对应的物品ID → ItemStaticData | `""` |
| canCombined | bool | string | 是否可合成 | `"false"` |
| combineParameter | int[] | string[] | 合成所需的物品ID列表 → ItemStaticData | `["1121","1701"]` |
| Describe | string[] | string[] | [中文详细描述, 英文详细描述] | `["沾湿的毛巾...","A damp towel..."]` |
| ShortDescribe | string[] | string[] | [中文简述, 英文简述] | `["沾湿的毛巾","Pungent Towel"]` |
| location | string[] | string[] | [中文位置名, 英文位置名] | `["罗莎的储藏室","RosaStorageRoom"]` |
| Chapter | string | string | 所属章节 | `"EPI01"` |
| folderPath | string | string | 美术资源文件夹 | `"EPI01\\RosaStorageRoom"` |
| desSpritePath | string | string | 详情大图路径 | `"SC101_item_114_big"` |
| mapSpritePath | string | string | 地图小图路径 | `"SC101_item_114"` |
| iconPath | string | string | 背包图标路径（可选） | `""` |
| Position | float[] | string[] | 场景中坐标 [X, Y, 可选Z] | `["739","1141"]` 或 `["1802","1042","-0.2"]` |
| ArtRequirement | string | string | 美术需求编号 | `"112"` |

**外键关联**：
- `analysedEvidence` → **ItemStaticData** 本表自引用
- `beforeAnalysedEvidence` → **ItemStaticData** 本表自引用
- `combineParameter[]` → **ItemStaticData** 本表自引用

**被引用**：
- **SceneConfig.ItemIDs[]** 引用此表
- **ExposeData.item[]** 引用此表
- **ExposeConfig.ExposeItemID[]** 引用此表
- **DoubtConfig.condition[].param** 当 type=1(道具) 时引用此表

**物品类型说明**（itemType 枚举）：

| 值 | 枚举名 | 含义 | 说明 |
|----|--------|------|------|
| 0 | decoration | 纯装饰 | 无法交互的场景装饰 |
| 1 | clue | 线索 | 现场痕迹类，拍照取证收集（不可分析/合成） |
| 2 | envir | 环境 | 环境叙事物品（可查看不可收集） |
| 3 | item | 物品 | 可拾取实物（含文件/信件/票据），`canAnalyzed/canCombined` 由物品自身性质决定 |
| 5 | door | 门 | 场景切换用的门 |
| 6 | interactive_item | 可交互道具 | 可交互但不可收集的道具 |
| 7 | embedded_item | 内嵌道具 | 嵌入场景中的道具 |
| 8 | upstairs | 上行楼梯 | 上行楼梯（场景切换） |
| 9 | downstairs | 下行楼梯 | 下行楼梯（场景切换） |

> **已废弃**：`itemType=4 note`——原"文件笔记"类型已合并入 `item`（用 `canAnalyzed=false` 表达不可分析）。NPC 口述信息走 `Testimony` 表（不属于 itemType）。

**分析机制**：
- 当 `canAnalyzed=true` 时，玩家分析该物品后会获得 `analysedEvidence` 指向的新物品
- 例：1114(沾湿的毛巾) → 分析后得到 1701(带有氯仿气味的毛巾)
- 新物品的 `beforeAnalysedEvidence` 反向指回原物品

**合成机制**：
- 当 `canCombined=true` 时，`combineParameter` 列出合成所需的原料物品ID

### 3.4 NPCStaticData — NPC基础信息表

### 3.5 NPCLoopData — NPC场景实例表

**用途**：NPC 在特定场景中的具体表现：绑定哪段对话、使用什么立绘、在场景中的位置。一个NPC在不同场景/轮次中有不同的 NPCLoopData 实例。

**主键**：`id`

**C# 类**：`Assets/Scripts/table/NPCLoopInfo.cs`

| 字段 | C#类型 | JSON类型 | 含义 | 示例值 |
|------|--------|----------|------|--------|
| id | int | string | 实例唯一ID | `"1001"` |
| NPC | NPCStaticData | object | NPC基础信息 → NPCStaticData表 | `{"id":"105",...}` |
| TalkInfo | Talk | object | 首次对话起始句 → Talk表 | `{"id":"105001001",...}` |
| LoopTalkInfo | Talk | object | 重复对话起始句 → Talk表 | `{"id":"105002001",...}` |
| ResPath | string | string | 待机立绘资源路径 | `"Art\Scene\NPC\EPI01\SC103_npc_Tommy1"` |
| ClickResPath | string | string | 点击后立绘资源路径 | `"Art\Scene\NPC\EPI01\SC103_npc_Tommy2"` |
| PosX | float | string | 场景中X坐标 | `"1101"` |
| Posy | float | string | 场景中Y坐标 | `"681"` |
| PosZ | float | string | 场景中Z深度（排序用） | `"-1"` |

**外键关联**：
- `NPC` → **NPCStaticData** 表（C# 通过 `_NPCId` 懒加载）
- `TalkInfo` → **Talk** 表（C# 通过 `_TalkInfoId` 懒加载）
- `LoopTalkInfo` → **Talk** 表（C# 通过 `_LoopTalkInfoId` 懒加载）

**被引用**：
- **SceneConfig.NPCInfos[]** 引用此表

**配置注意事项**：
- JSON 中 NPC、TalkInfo、LoopTalkInfo 是完整的内联对象（冗余存储），但 C# 运行时通过 ID 懒加载查表
- TalkInfo 是玩家首次与该NPC对话时触发的对话链
- LoopTalkInfo 是再次对话时的内容（较短）
- 坐标系：(0,0) 在左上角，2560x1440 分辨率为基准

### 3.6 Talk — 对话配置表

**用途**：单句对话的完整定义，包含说话者、台词、分支逻辑、视频资源引用。对话通过 `next` 字段形成链式结构。

**主键**：`id`

**C# 类**：`Assets/Scripts/table/Talk.cs`

| 字段 | C#类型 | JSON类型 | 含义 | 示例值 |
|------|--------|----------|------|--------|
| id | int | string | 对话ID | `"105001001"` |
| step | int | string | 在当前对话组中的步骤序号 | `"1"` |
| isRight | bool | string | 是否显示在右侧（区分对话双方） | `"false"` |
| waitTime | float | string | 显示前等待时间（秒） | `"0"` |
| Speaker | NPCStaticData | object | 说话者 → NPCStaticData表 | `{"id":"105",...}` |
| Location | string[] | string[] | [中文地点, 英文地点]（对话发生场所） | `["Tommy办公室 - 11月4日凌晨","TommyOffice - November 4th, Early Morning"]` |
| Words | string[] | string[] | [中文台词, 英文台词] | `["谁？...","Who?..."]` |
| next | int | string | 下一句对话ID → Talk表 | `"105001002"` |
| script | DialogueAction(enum) | string | 动作脚本类型 | `""` 或 `"expose"` |
| Parameters | TalkParam[] | object[] | 动作参数数组 | 见下方 |

### 3.9 DoubtConfig — 疑点配置表

**用途**：玩家在调查中产生的疑问。每个疑点有解锁条件，当条件满足后疑点显示给玩家，引导调查方向。

**主键**：`id`

**C# 类**：`Assets/Scripts/table/DoubtConfig.cs`

| 字段 | C#类型 | JSON类型 | 含义 | 示例值 |
|------|--------|----------|------|--------|
| id | int | string | 疑点ID | `"1101"` |
| isFragment | bool | bool | 是否为疑点碎片（默认 false）| `false` |
| condition | DoubtCondition[] | object[] | 解锁条件列表（需全部满足） | 见下方 |
| text | string | string | 疑点/碎片文本（中文） | `"Rosa当晚究竟在哪工作？"` |

**isFragment 语义**：
- `false`（默认）= 完整疑点：condition 满足 → 显示为疑点，当前目标时计入 clearDoubts
- `true` = 疑点碎片：condition 满足（通常单件） → 显示为灰调"碎片"UI，**不计入 clearDoubts**
- 碎片等待另一件证据/证言拼齐后，对应 `isFragment=false` 的完整疑点 condition 全满足，碎片自动隐藏
- 设计规则：禁止"碎片 + 碎片 = 疑点"三层合并；合并只发生一次（由策划/设计侧保证）
- 详见 `docs/游戏系统/核心玩法/疑点系统.md` 第八节

**DoubtCondition 子结构**：

| 字段 | C#类型 | JSON类型 | 含义 |
|------|--------|----------|------|
| type | DoubtConditionType(enum) | string | 条件类型 |
| param | int | string | 条件参数（对应物品ID/关系网ID/时间线ID） |

**疑点条件类型**（DoubtConditionType 枚举）：

| 值 | 枚举名 | 含义 | param 含义 |
|----|--------|------|------------|
| 1 | Item | 道具 | ItemStaticData.id — 需要拥有该道具 |
| 2 | RelationNetwork | 关系网 | TestimonyItem.id — 需要建立该关系连接 |
| 3 | Timeline | 时间线 | TestimonyItem.id — 需要在时间线上标记该事件 |

**被引用**：
- **ChapterConfig.doubts[]** 包含此表数据

**JSON 示例**：
```json
{
  "id": "1101",
  "condition": [
    {"type": "3", "param": "1031002"},
    {"type": "1", "param": "1115"}
  ],
  "text": "Rosa当晚究竟在哪工作？"
}
```
→ 当玩家在时间线上标记了证词道具 1031002 **且** 拥有道具 1115 时，疑点"Rosa当晚究竟在哪工作？"被解锁

### 5.2 懒加载属性模式

C# 数据类使用**私有ID字段 + 公开属性 getter** 实现关联表的按需加载：

```csharp
// SceneConfig.cs 中：
private int[] _NPCInfosIds;              // JSON序列化时存储的是ID数组

public NPCLoopData[] NPCInfos {          // 外部访问时按需查表
    get {
        if (_NPCInfosIds == null) return null;
        var result = new NPCLoopData[_NPCInfosIds.Length];
        for (int i = 0; i < _NPCInfosIds.Length; i++)
            result[i] = TableConfig.GetTableByKey<NPCLoopData>(
                (TableEnum)14, _NPCInfosIds[i]);
        return result;
    }
}
```

这种模式出现在所有包含外键引用的表中，包括：
- SceneConfig：`_locationId` → `location`，`_NPCInfosIds` → `NPCInfos`
- NPCLoopData：`_NPCId`/`_TalkInfoId`/`_LoopTalkInfoId` → `NPC`/`TalkInfo`/`LoopTalkInfo`
- Talk：`_SpeakerId` → `Speaker`
- ChapterConfig：`_doubtsIds`/`_exposesIds` → `doubts`/`exposes`
- Testimony：`_npcId`/`_evidenceItemIds` → `npc`/`evidenceItem`
- GameFlowConfig：`_deceasedId` → `deceased`

## 六、对话系统详解

### 6.1 对话链结构

对话是链式结构，每个 Talk 条目代表一句对话：

```
Talk[105001001] ──next──→ Talk[105001002] ──next──→ Talk[105001003] ──next──→ ...
  Tommy:"谁？..."        Zack:"我是Zack..."      Tommy:"楼下出了大事..."
```

### 6.2 对话分支

当 `script="1"` (branches) 时，对话暂停等待玩家选择：

```
Talk[102003024] (script=branches)
  ├── "良心是奢侈品" → Talk[102003401]
  └── "我很有经验"   → Talk[102003402]
```

分支信息存储在 `Parameters[]` 数组中：
- `ParameterStr` = 选项显示文本
- `ParameterInt` = 选择后跳转的 Talk ID

### 6.3 指证触发

当 `script="expose"` / `"Expose"` 时，标记当前对话为"可指证点"。此时游戏UI允许玩家出示证据反驳该句话。

### 6.4 特殊动作

| script值 | 行为 |
|----------|------|
| `""` (空/none) | 正常播放，自动跳 `next` |
| `"1"` (branches) | 显示分支选项 |
| `"end"` | 对话结束 |
| `"get"` | 获得道具（ParameterInt=物品ID） |
| `"get_Evidence_Box"` | 获取证据箱 |
| `"interrupt"` | 对话中断 |
| `"Unlock_Cabaret_Hall"` | 解锁歌舞厅 |
| `"expose"` | 可指证标记 |
| `"change_scene"` | 切换场景（ParameterInt=sceneId） |
| `"unlock_map"` | 解锁地图 |
| `"play_video"` | 播放全屏转场视频（ParameterStr=视频路径, 第二个ParameterInt=可选切场景ID），播完继续next |
| `"finalexpose"` | 最终指证 |

### 6.5 视频关联

每句对话关联一个数字人口播视频：

```
Assets/Resources/{PathMgr.VideoPrePath}/
  └── EPI01/           (videoEpisode)
      └── loop1/       (videoLoop)
          └── tommy_001/  (videoScene)
              └── 105001001.mp4  (videoId)
```

## 七、指证系统详解

### 7.1 指证流程

```
1. 进入指证场景（ChapterConfig.initScene → SceneConfig）
2. 显示指证UI（ExposeConfig 提供NPC信息和可用物品池）
3. 播放指证对话链（Talk 链式播放）
4. 遇到 script="expose" 的对话 → 玩家可出示证据
5. 匹配 ExposeData 方案：
   - 比对玩家出示的 testimony + item 组合
   - 匹配成功 → 跳转 ExposeData.talkId 继续对话
   - 匹配失败 → 提示错误
6. 所有回合完成 → 本轮指证成功
```

### 7.2 数据结构关系

```
ChapterConfig (id=101, Loop1)
├── exposeNpcId = 103 (Rosa)
├── topBg / bottomBg (指证场景背景)
└── exposes[] = [
    ExposeData(id=1): testimony=1031002 + item=[1115] → talkId=110001  (第1轮)
    ExposeData(id=2): testimony=1031002 + item=[1701,1121] → talkId=110003  (第2轮)
    ExposeData(id=3): testimony=1031002 + item=[1702] → talkId=110007  (第3轮)
    ExposeData(id=4): testimony=1031002 + item=[] → talkId=110015  (最终轮，无需物品)
]
```

### 7.3 疑点系统

疑点(Doubt)是引导玩家调查方向的问题，需要满足特定条件才能解锁：

```
DoubtConfig(id=1101): "Rosa当晚究竟在哪工作？"
  condition = [
    {type=3(时间线), param=1031002},  // 需要在时间线上标记"Rosa从九点半开始在地下室打扫"
    {type=1(道具), param=1115}         // 需要拥有道具1115
  ]
```

当所有 condition 满足时，疑点显示给玩家。疑点与指证的关系：
- 疑点引导玩家收集证据
- 收集到足够证据后进入指证环节
- 指证时使用的物品与解锁疑点所需的物品高度重合

## 八、JSON 格式注意事项

### 8.1 数据类型差异

JSON 中所有值都是**字符串**形式，C# 在反序列化时自动转换：

| JSON | C# | 示例 |
|------|-----|------|
| `"105"` | int | NPC ID |
| `"true"` / `"false"` | bool | canAnalyzed |
| `"0.5"` | float | waitTime |
| `["中文","English"]` | string[] | 双语文本 |
| `"1"` | enum | 枚举值（通过 (enumType)int 转换） |

### 8.2 已知 JSON 格式问题

游戏导出的 JSON 有三个常见问题（Preview 页面的 `fixJson()` 函数处理这些问题）：

1. **字符串中的字面换行符**：多行描述中的 `\n` 未转义
2. **路径中的反斜杠**：`Art\Scene\...` 需转义为 `Art\\Scene\\...`
3. **字符串中的未转义双引号**：描述文本中嵌入的 `"` 未转义

### 8.3 内联 vs 引用

JSON 中外键对象采用**完整内联**方式（冗余存储），例如 SceneConfig 的 NPCInfos 中会内联完整的 NPC 和 Talk 对象。但 C# 运行时通过二进制序列化只存储 ID，通过懒加载 getter 查表获取实际对象。
