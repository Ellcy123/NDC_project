# 疑点系统

## 一、核心概念

**疑点（Doubt）** 与 **疑点碎片（Fragment）** 是玩家主动整理信息后发现的矛盾点，也是进入指证的前置条件。

- **疑点**：单循环内可独立成立的矛盾点。condition 可以是 1 件或多件线索（道具 / 证词 / 关系网 / 时间线）。
- **疑点碎片**：跨循环才能拼齐的矛盾点的中间态。**2~3 个碎片合并成一个疑点**。

每个疑点 / 碎片都有 **出现（Appear）** 与 **解决（Resolve）** 两个阶段。

**设计哲学**：系统在后台做矛盾检测，玩家只需做"拿到物品/拖拽到面板/化验合成"等操作。操作量高但思考负担低——玩家感觉自己在推理，但实际不会卡关。

---

## 二、两态生命周期

### 出现（Appear）——玩家侧触发
condition 全部满足时，疑点 / 碎片点亮，并计入当前 Loop 的收集进度条。

### 解决（Resolve）——系统侧触发
**进入下一个 Loop 时，当前 Loop 所有已出现的疑点统一翻页标记为"已解决"**（视觉灰化，归档至历史记录面板，不再计入任何进度）。

### 跨 Loop 碎片的例外
**已出现但尚未合并成疑点的碎片不随 Loop 翻页**——它持续存在，直至被合并成疑点。只有合并后的疑点，才按其所在 Loop 的翻页规则被解决。

---

## 三、获取途径

condition 可由以下五种途径满足：

| # | 途径 | 操作 | 示例 |
|---|------|------|------|
| ① | 整理人物关系 | 将证词拖到 NPC 头像 | 发现 Vivian 和 Webb 的关系存疑 |
| ② | 整理时间线 | 将证词拖到时间线面板 | Rosa 的时间线与死亡时间重叠 |
| ③ | 获取物品 | 场景探索/对话获得关键物品 | 获得带血的工作记录卡 |
| ④ | 分析物品 | CASE BOARD 化验/指纹分析 | 毛巾上检测出氯仿残留 |
| ⑤ | 合成物品 | CASE BOARD 合并两个物品 | 医疗账单+工作记录→动机 |

## 四、DoubtConfig 字段

```json
{
  "id": "疑点ID",
  "isFragment": false,
  "condition": [
    {"type": "条件类型", "param": "目标ID"}
  ],
  "text": "疑点文本"
}
```

| 字段 | 说明 |
|------|------|
| `id` | `{unit}{loop}{seq}`，例 9502 = Unit9 Loop5 第 2 项。碎片与疑点共用编号空间。 |
| `isFragment` | `true` 为碎片，`false` 为疑点。旧 Unit 不填默认 false。 |
| `condition` | condition 数组，**全部满足才出现**。 |
| `text` | 面板显示文本。 |

### condition.type 枚举

| type | 名称 | param 含义 |
|------|------|-----------|
| 1 | **Item（道具）** | 需拥有 ItemStaticData 中的指定物品 |
| 2 | **RelationNetwork（关系网）** | 需在关系网面板建立指定连接 |
| 3 | **Timeline（时间线）** | 需在时间线面板标记指定证词 |

## 五、疑点与碎片

### 疑点（isFragment=false）
- `condition` 可为 1~N 件（单循环内能拼齐即可）
- 全部条件满足 → 出现

### 碎片（isFragment=true）
- `condition` 为单件
- 条件满足 → 出现，显示为灰调 + "碎片"角标，等待合并

### 合并规则

- **2~3 个碎片 → 1 个疑点**
- 父疑点的 `condition` = 各子碎片 `condition` 的并集
- 运行时：当父疑点的 condition 全部满足、且某碎片的 condition 是父疑点 condition 的子集 → 父疑点点亮，子碎片从面板隐藏（进入父疑点卡片内部作为线索列表）
- **禁止三层结构**：碎片 + 碎片 = 疑点（允许），碎片 + 碎片 = 更大的碎片（禁止）
- **证据 ↔ 疑点/碎片 是一对一关系**：每件道具 / 证词只能挂在**唯一一个**疑点或碎片的 condition 里，不能被多个疑点同时引用
  - 设计时如果发现一件证据"好像哪个疑点都能挂"，说明该证据承载了多重推理价值——必须在证言层面拆成多条独立证言，或在物证层面拆成多件物证，各自挂到不同疑点
  - 指证轮次的**使用关系**不受此约束：同一件证据在多个指证轮次、不同谎言维度下作为击穿件出示是允许的

---

## 六、指证硬约束

**每一个指证步骤（ExposeStep）所需的道具 / 证词，必须至少被某个疑点或碎片的 `condition` 引用。**

> 不存在"只在指证时登场、但没进入任何疑点/碎片"的游离道具。指证前先反查：这件东西被哪个疑点/碎片装载？

**进入指证门槛**：当前 Loop 的疑点 + 碎片收集度 = **100%** 才能点开 EXPOSE。

### 时序硬约束（防止流程卡死）

**某件证据 X 在 Loop N 的指证中被使用 → X 所属的疑点 / 碎片必须出现在 Loop N 或更早**。

等价表述：疑点 / 碎片的出现 Loop ≤ 它 condition 里任何证据的**最早指证使用 Loop**。

**为什么必须**：
1. 玩家进入 Loop N 指证前需 100% 收集 Loop N 的疑点 + 碎片——这是他们的"导览图"
2. 若 X 的所属疑点在 L > N 才出现，那么在 Loop N 阶段玩家的导览图里没有任何指引去收集 X
3. 玩家可能跳过 X 所在的场景 / 对话直接进指证 → 指证缺证据无法击破 → Loop 已翻页无法回头 → **流程卡死**

**设计检查清单**：
- 列出每个疑点 / 碎片的出现 Loop（记为 L_doubt）
- 列出 condition 里每件证据在指证里被使用的 Loop（记为 L_use，可能有多个 Loop 使用同一证据）
- 对每件证据：必须 `L_doubt ≤ min(L_use)`
- 等价：把"证据第一次出现在指证里的 Loop"作为该证据挂载疑点的最晚时间窗口

**典型违例**：一件证据在 L4 第一次被获取、在 L4 就用于指证击破；设计者把它挂到 L5 的疑点 condition 里——违反规则。

## 七、收集进度百分比

**分母** = 当前 Loop 新增的「疑点 + 碎片」总数
**分子** = 其中已出现的数量
**进度** = 分子 / 分母 × 100%

> 例：Loop1 有 4 个疑点 + 1 个碎片 = 5 项，每找到 1 项 = 20%。

**跨 Loop 持久的碎片不计入新 Loop 的分母**——它们是上个 Loop 留下的"历史遗留"，只在原 Loop 的进度条里出现过；合并完成时直接点亮父疑点，不影响当前 Loop 的收集节奏。

## 八、UI 表现

### 右上角进度条
```
┌──────── Loop1 疑点进度 ────────┐
│ ▓▓▓▓░░░░░░░░░░░░░░░ 20% (1/5)  │
└────────────────────────────────┘
```

### EXPOSE 按钮
- < 100% → 灰色，提示"收集更多疑点后可进行指证"
- = 100% → 红色，可点击进入指证

### 新疑点 / 碎片弹窗
获得新条目时左下角弹出：
```
┌──────────────────────┐
│ ！获得新疑点          │
│ Rosa 的时间线存疑     │
└──────────────────────┘
```
碎片专用样式：标题改为「获得新线索（碎片）」，灰调配色。

### 翻页动画
进入下一个 Loop 时，当前 Loop 的疑点卡片执行"归档"动画（灰化 + 缩小 + 飞入历史面板）。跨 Loop 碎片保留原位，不参与动画。

## 九、完整实例

### 例 1：Loop1 单循环疑点组（4 疑点 + 0 碎片）

#### 疑点 1101：Rosa 当晚究竟在哪工作？
```json
{"id":"1101","isFragment":false,
 "condition":[{"type":"3","param":"1031002"},{"type":"1","param":"1115"}],
 "text":"Rosa当晚究竟在哪工作？"}
```
**路径**：Rosa 对话 → 证词 1031002 → 拖入时间线；Tommy 办公室 → 物品 1115 排班表。两条件满足 → 出现。

#### 疑点 1102：这条毛巾为什么有奇怪的味道？
```json
{"id":"1102","isFragment":false,
 "condition":[{"type":"1","param":"1701"},{"type":"1","param":"1121"}],
 "text":"这条毛巾为什么有奇怪的味道？"}
```
**路径**：物品 1114 毛巾 → 化验得 1701 氯仿气味毛巾；获取 1121 清洁记录。

#### 疑点 1103：是谁拖得我？
```json
{"id":"1103","isFragment":false,
 "condition":[{"type":"1","param":"1702"}],
 "text":"是谁拖得我？"}
```
**路径**：合成 1701 + 1121 → 1702。

#### 疑点 1104：11 点 30 分的枪声是案发时间吗？
```json
{"id":"1104","isFragment":false,
 "condition":[{"type":"3","param":"1051003"}],
 "text":"11点30分的枪声是案发时间吗？"}
```
**路径**：Tommy 证词 1051003 → 拖入时间线。

> 四个疑点中每件 condition 引用的道具 / 证词（1031002, 1115, 1701, 1121, 1702, 1051003）都必须被 Loop1 的指证步骤用到——这是第六节的硬约束。

### 例 2：跨 Loop 碎片 → 疑点（L2 + L5）

```json
// L2 碎片
{"id":"9201","isFragment":true,
 "condition":[{"type":"3","param":"9021001"}],
 "text":"Jimmy 向 Anna 借过钱（碎片）"}

// L5 碎片
{"id":"9501","isFragment":true,
 "condition":[{"type":"3","param":"9051004"}],
 "text":"Anna 怀孕（碎片）"}

// 合并后的疑点（挂 L5）
{"id":"9502","isFragment":false,
 "condition":[
   {"type":"3","param":"9021001"},
   {"type":"3","param":"9051004"}
 ],
 "text":"Jimmy 的经济动机？"}
```

**运行时**：
- L2 获得证词 9021001 → 碎片 9201 出现（计入 L2 进度 20%，假设 L2 共 5 项）
- L3、L4 碎片 9201 持续存在，不翻页
- L5 获得证词 9051004 → 碎片 9501 condition 满足 → 同时父疑点 9502 condition 全满足 → **9502 点亮，9201 与 9501 自动隐藏进 9502 卡片内部**
- 9502 计入 L5 进度（作为 L5 新出现的疑点 1 项）

## 十、向后兼容

Unit1 / Unit3 等旧 Unit 的 DoubtConfig 可不填 `isFragment`（默认 false），全部视作单循环疑点。进度条分母 = 该 Loop 全部疑点数，指证门槛 = 100%。旧数据无需回改。
