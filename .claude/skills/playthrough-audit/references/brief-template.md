# Loop Audit Brief 模板

每个 Loop 生成一份 brief，写入 `audit-reports/{Unit}_{date}/briefs/L{N}_brief.md`。

Brief 的目标是让审计 agent **不需要读原始 state/config 文件就能完成审计**。
对话原文太长，只放摘要 + 路径（供需要逐句审查的 agent 按需读取）。

## 填写指南

### 从哪里提取数据

| Brief 字段 | 数据来源 |
|------------|---------|
| 已知事实 | state yaml 的 `known_facts` 字段 |
| 已持有证据 | 前序 Loop 的证据 + ItemStaticData |
| 场景列表 | state yaml 的 `scenes` 字段 + SceneConfig |
| 信息点分配 | state yaml 的 NPC `knowledge_to_share` / `active_topics` |
| 新增证据 | state yaml 的搜证配置 + ItemStaticData |
| 新增证词 | state yaml 的 testimony 配置 + TestimonyItem |
| 疑点 | DoubtConfig 中本 Loop 可解锁的 |
| 指证 | ExposeData + ExposeConfig + state yaml 的 expose 段 |
| 对话摘要 | **必须读取对话 JSON 原文**（见下方验证规则） |
| NPC 知识边界 | state yaml 中每个 NPC 的 knowledge 段 |
| 前后衔接 | state yaml 的 narrative 段 + 前后 Loop 的 state |

### 对话摘要验证规则（必须遵守）

Brief 中的"§6 对话摘要"**禁止**仅从 state 文件/设计文档抄写描述。必须：

1. **读取每个对话 JSON 的前 5-10 条 `cnWords`**，确认场景实际内容与 state 描述一致
2. 如果实际内容与 state 描述不一致（例如 state 说"开篇引导"但 JSON 内容是 post-expose 讨论），以 **JSON 实际内容为准**
3. 在摘要中标注该对话发生的**叙事时间点**：指证前(pre-expose) / 指证后(post-expose) / 与指证无关
4. 对话的"功能"描述必须基于实际台词内容，不能照搬 state 文件的规划标注

**反面案例**：state 文件标注 emma_004 为"开篇引导，汇报 Rosa 供词"，但 JSON 实际内容是 Expose 后讨论 Whale 策略。Brief 照抄 state 描述 → 后续 agent 误判为"开篇破梗" → Critical 误报。

### 长度控制

- 每份 brief 控制在 **2000-3000 字**
- 对话摘要是最长的部分，每个 NPC 控制在 300 字以内
- 证据/证词表格行数 = 实际数量，不做截断
- 如果一个 Loop 内容特别丰富，可以适当超出但不超过 4000 字
