# 对话草稿独立审查

独立于生成流程的 Critic Agent。在全新上下文中审查对话草稿，捕捉生成时的盲点。

---

## 使用方式

用户调用时指定循环编号，例如：`/review-dialogue 1`（审查 Loop 1 的对话草稿）。

如果用户未指定循环编号，询问要审查哪个循环（1-6）。

---

## 输入文件

执行前必须读取以下文件。**全部读完后再开始审查，不要边读边审。**

### 1. 对话草稿（审查对象）

按优先级查找：
1. `AVG/0304/对话草稿/Loop{N}/` 目录下所有 `.md` 文件（每个 NPC 一个文件）
2. `AVG/对话配置工作及草稿/生成草稿/Loop{N}_生成草稿.md`（合并草稿）
3. `AVG/对话配置工作及草稿/Loop{N}_对话草稿.md`（旧版草稿）

如果同时存在多个来源，优先审查第 1 项（单文件目录），因为这是最终的工作版本。

### 2. State 文件（对照基准）
`AVG/0304/state/loop{N}_state.yaml`

这是"设计蓝图"——对话必须忠实实现 state 中定义的内容。

### 3. 知识池
`AVG/对话配置工作及草稿/前置配置/npc_knowledge_pools.yaml`

重点提取每个 NPC 的 `blind_spots` 列表。

### 4. 角色人设
`AVG/对话配置工作及草稿/前置配置/characters/{出场NPC}.md`

只读本循环出场的 NPC 的人设文件。从 state 文件的 `opening` 和 `scenes` 段落确定出场 NPC 列表。

### 5. 数据表
- `preview_new2/data/table/ItemStaticData.json`（证据 ID、名称、描述）
- `preview_new2/data/table/TestimonyItem.json`（证词 ID、内容）

### 6. 主角人设
- `剧情设计/00_世界观与角色/主角设计/Zack_Brennan.md`
- `剧情设计/00_世界观与角色/主角设计/Emma_OMalley.md`

### 禁止读取

**绝对不要**读取 `.claude/commands/generate-dialogue.md`。审查必须独立于生成逻辑，避免"自审自"偏差。

---

## 审查清单（12 项）

### 准备工作

读完所有输入文件后，先为本循环建立以下索引（不输出，内部使用）：

**A. NPC 禁区索引**：对每个 NPC，合并 `blind_spots`（知识池）+ `withheld_topics`（state 文件）→ 形成该 NPC 的完整禁区清单。

**B. 证据/证词索引**：从 state 文件提取所有 `testimony_ids`，标记哪些是 `⚠谎言`、`⚠偏见`。

**C. 信息所有者索引**：从 state 文件的 NPC 条目中，确定每条关键信息的"信息所有者"（哪个 NPC 负责传达）。

**D. 玩家已知信息索引**：`player_context.known_facts` + opening 中获得的信息 = Zack 在自由探索阶段开始时已知的全部信息。

---

### 1. 知识边界（blind_spots / withheld_topics 泄漏检查）

**检查方法**：
- 对每个 NPC 的**每一句台词**，逐条对照该 NPC 的禁区清单
- 检查**直接泄漏**：NPC 说出了禁区中的具体信息（人名、数字、地点、结论）
- 检查**间接暗示**：NPC 的措辞虽未直说，但玩家可以推断出禁区信息
  - 例：blind_spot 是"Rose藏在地下室" → NPC 说"地下室最近改装过" = 间接暗示
  - 例：blind_spot 是"月息12%" → NPC 说"利息翻了好几倍" = 合法模糊表述（PASS）
- 检查 NPC 是否泄漏了本循环 Expose 的结论
- 检查 NPC 是否泄漏了后续循环才揭示的信息

**判定**：有任何直接泄漏 → FAIL。间接暗示视严重程度标注为 WARNING 或 FAIL。

### 2. 信息分配

**检查方法**：
- 提取每个 NPC 对话中传达的**核心新信息**（不包括寒暄、情绪、重复已知事实）
- 检查同循环内不同 NPC 是否传递了**相同的核心信息**
- 每条信息应只有一个"信息所有者"

**判定**：同一条核心信息出现在两个 NPC 的对话中 → FAIL。

### 3. 证据获取覆盖

**检查方法**：
- 从 state 文件提取所有 NPC 的 `testimony_ids` 列表
- 在对话草稿中搜索所有 `get` 标记（格式：`### {ID} \`get\` → {证据ID}`）
- 逐一交叉核对：
  - state 中的每个 testimony_id 是否都有对应的 `get` 句？
  - 对话中的每个 `get` 是否都在 state 的 testimony_ids 中？

**判定**：缺少 `get` 或多出 `get` → FAIL。

### 4. 实物引用准确性

**检查方法**：
- 找到对话中**提及物品/证据**的句子（尤其是 Expose 部分）
- 在 `ItemStaticData.json` 中查找对应 ID 的 `Describe` 和 `ArtRequirement` 字段
- 检查对话中的物理描述是否与数据表一致
  - 例：数据表写"橡木拐杖，握柄刻字'F.K.'"，对话写"铁拐杖" → FAIL

**判定**：描述不一致 → FAIL。

### 5. 角色语气

**检查方法**：
- 对每个 NPC，对照 `characters/{npc}.md` 中的"说话方式"和"绝对不会做的事"
- 检查是否有"串味"（NPC A 说了 NPC B 风格的话）
- 检查 `[方括号]` 内容：只允许情绪/表情标注（如 `[皱眉]`、`[冷笑]`）
  - 禁止：`[内心独白]`、肢体动作描述（如 `[站起来走到窗前]`）
  - 方括号内的**短小动作提示**可以接受（如 `[弹了弹烟灰]`），但不应写长句
- 检查 Zack 的对话是否体现了人设特征（克制、自嘲、不说教）

**判定**：严重串味或格式违规 → FAIL。轻微语气偏差 → WARNING。

### 6. 对话链完整性

**检查方法**：
- 提取所有 `### {ID}` 对话节点
- 对每个 `branches` 节点：
  - 检查所有选项的跳转目标 ID 是否存在
  - 检查每个分支路径是否有 `→ 汇合至 {ID}` 标记
  - 检查汇合目标 ID 是否存在
- 对每个 Talk/Expose 段落：
  - 检查是否以 `end` 标记结尾
- 检查是否有孤立节点（ID 存在但没有任何路径到达）

**判定**：断链、缺少汇合、缺少 end → FAIL。

### 7. 获取节奏

**检查方法**：
- 在每个 Talk 段落中，按顺序找到所有 `get` 标记
- 计算连续两个 `get` 之间的**对话行数**（`### {ID}` 的数量）
- 如果分支中有 `get`，只计算同一路径上的间距

**判定**：间距 < 3 → FAIL。间距 = 3 → WARNING（建议扩展）。

### 8. ID 编码

**检查方法**：
- 所有 `### {ID}` 中的 ID 必须是 **9 位数字**（Talk）
- 格式：`{loop}{scene}{sequence}`
  - loop: 与文件所属循环一致
  - scene: 与 state 文件中的场景 ID 后两位一致
  - sequence: 3 位序号，分支时千位变化（❶=1xx, ❷=2xx, ❸=3xx）
- 检查文件内是否有重复 ID

**判定**：格式错误或重复 → FAIL。

### 9. Expose 谎言驱动

**检查方法**：
- 对每个 `Lie` 标记：
  - 谎言是否由**嫌疑人主动说出**（止损式退守）？
  - 还是被 Zack 喂话后被动否认？（后者 → FAIL）
- `🎯 可用证据` 中的证据 ID 是否在 `ItemStaticData` 中存在？
- Round 1 的 Lie 是否对应 Talk 中已收集的 `⚠谎言` 证词？
- 后续 Round 的 Lie 是否是嫌疑人在被揭穿后的**新谎言**？

**判定**：Zack 喂话、证据 ID 无效、结构不符 → FAIL。

### 10. 鉴赏力节点

**检查方法**：
- `source` 类型：
  - 是否有 HTML 注释 `<!-- 后面要考：{内容} -->`？
  - source 信息是否自然嵌入对话（不是为了考而硬塞）？
- `quiz` 类型：
  - 是否有 HTML 注释 `<!-- 鉴赏力 #{编号} | {领域} | 信息来源: {出处} -->`？
  - 是否是 **NPC 考 Zack**（不是 Zack 考 NPC）？
  - 是否有 3 个选项（1 正确 + 2 迷惑）？
  - 迷惑选项是否"看起来有道理"？（禁止"不知道""还在查"等逃避式选项）
  - 正确路径是否有 NPC 认可 + 信息展开？
  - 错误路径是否有 NPC 冷淡反应？
  - 三条路径是否汇合？
- source↔quiz 配对：
  - 每个 source 是否有对应的 quiz？（可能在后续循环）
  - 是否有同 NPC 同循环自考自？
  - quiz 考的信息是否有足够的"信息距离"（不能刚说完就考）？

**判定**：缺少注释、结构不完整、逃避式选项、自考自 → FAIL。

### 11. 玩家视角信息传达

**检查方法**：

逐句检查，应用以下四个测试：

**测试 A："去掉玩家"检验** — 去掉玩家这个观察者，角色还会说这句话吗？如果不会，这句话就是在给玩家灌信息。
- ✗ "我们现在应该去搜证了"（系统状态自述）
- ✓ "你再进去一趟，这次当成案发现场来搜"（Emma 有动机说这话）

**测试 B："As You Know, Bob"检验** — 角色是否在互说双方都已知的信息？
- ✗ Morrison 对 Emma 说"你知道的，蓝月亮案之后 Brennan 就盯上我了"（双方都知道）
- ✓ Morrison 对 Zack 说"蓝月亮那位"（合理的嘲讽）

**测试 C：Zack 转述检验** — Zack 是否转述了玩家未目睹的事件？
- ✗ Zack 说"我听说 Vinnie 去了码头"（玩家没看到这个场景）
- ✓ Zack 说"我在码头看到了 Vinnie"（Zack 亲身经历）
- 如果 NPC 在场，让 NPC 自己说出信息，不要让 Zack 代述

**测试 D：独白检验** — `[内心独白]` 是否作为独立对话行出现？
- 游戏中每行 `> 文本` 都渲染为角色台词，内心想法不应显示为对话句
- 如需标注 Zack 的想法，用 `<!-- 【Zack内心解读】... -->` HTML 注释

**判定**：任何测试失败 → FAIL，并注明是哪个测试。

### 12. 平行场景信息隔离

**检查方法**：
- 从 state 文件确定哪些场景是 🔓 自由探索场景（它们之间是平行关系）
- 在每个自由探索场景的对话中，检查 Zack 是否引用了**同循环其他自由探索场景**才能获得的信息
- **可引用的信息来源**：
  - ✓ `player_context.known_facts`（前序循环累积）
  - ✓ 本循环 Opening 中获得的信息
  - ✓ 前序循环中该 NPC 自己说过的话
- **不可引用**：
  - ✗ 同循环其他自由探索场景中获得的证据/证词
  - ✗ 同循环其他 NPC 说的话（除非是 Opening 中的）
- Expose 场景**例外**：Expose 可以引用本循环所有场景的信息

**判定标准**：
- 对每句 Zack 的台词自问："如果玩家**第一个**就来了这个场景，Zack 能说这句话吗？"
- 如果不能 → FAIL。

## 输出格式

审查完成后，输出结构化报告：

```markdown
# 对话审查报告: Loop {N}

## 总览
- 通过: X/12 项
- 未通过: Y/12 项
- 警告: Z 项
- 问题总数: W

## 详细结果

### 1. 知识边界 | ✅ PASS / ❌ FAIL
[若 FAIL:]
- **{NPC中文名}** ID `{对话ID}`: "{问题台词}"
  - 违反: {blind_spot 或 withheld_topic 原文}
  - 严重度: 直接泄漏 / 间接暗示
  - 建议: "{修改建议}"

### 2. 信息分配 | ✅ PASS
（无问题）

... （依次列出 12 项）

## 必须修复（影响游戏逻辑或玩家体验）
[汇总所有 FAIL 项中的关键问题]

## 建议改进（润色优化）
[汇总 WARNING 和非关键建议]
```

如果用户要求保存报告，输出到：`AVG/0304/对话草稿/Loop{N}/review_report.md`
