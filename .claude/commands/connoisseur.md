# 鉴赏力节点设计与写入

为指定章节/循环设计或审查鉴赏力（Connoisseur）节点。覆盖：state文件写入、对话草稿写入、节点审查。

---

## 使用方式

用户可能以以下方式调用：
- `/connoisseur design Unit3 L3` — 为Unit3的L3设计鉴赏力节点
- `/connoisseur review Unit3` — 审查Unit3所有鉴赏力节点
- `/connoisseur write L3` — 将已设计的节点写入state文件
- `/connoisseur` — 进入交互模式，询问用户要做什么

## 鉴赏力系统规则速查

### 系统机制

- **三选一节点**：对话分支形式，不是考题
- **选对**：NPC态度积极，多说1-3句世界观/时代知识（成就分值+1）
- **选错**：NPC冷淡/敷衍，1句带过（不扣分）
- **"我先想想"**：允许暂时离开（高压场景除外）
- 选对加分、选错不扣分、不存在信任度数值

### 信息来源限定

只有两种合法来源：
1. **对话** — 当前Loop或之前Loop中NPC已说过的内容
2. **证据/道具** — 已收集的物品描述、场景环境道具

写Loop N的quiz时，只能引用Loop 1~N的信息。

### 信息距离

- source和quiz之间**至少跨场景**，理想跨循环
- **禁止**同一段对话"刚说完就考"
- 理想：Loop N测试Loop 1~N-1的旧信息，或当前Loop其他场景已给出的信息

### 节点编号

| 章节 | 编号范围 |
|------|---------|
| EPI01 (Unit1) | #201-#299 |
| EPI02 (Unit2) | #001-#099 |
| EPI03 (Unit3) | #101-#199 |

### 每Loop密度

| 阶段 | 节点数 | 比例 |
|------|-------|------|
| L1-L2 | 2-3 | source为主 |
| L3-L4 | 2-3 | quiz为主 |
| L5-L6 | 1-2 | quiz为主 |

每章总计不超过15个。

## 三大设计原则

### 原则1：主语是Zack不是NPC

**Zack主动展示知识 → NPC被动反应。** 不是NPC提问Zack回答。

- ❌ Tony问"我的招牌鸡尾酒叫什么？"
- ✅ Tony说"来一杯？"，Zack选"来杯Southside"

### 原则2：NPC必须有角色动机

NPC关心Zack是否知道某事，必须有角色层面的理由，不是"系统需要考试"。

### 原则3：服务对话推进而非打断

节点是对话的分水岭——Zack展示的水平决定NPC接下来说多少、说多深。

### 原则4：正反馈优先——让玩家容易选对

**鉴赏力的目标是让玩家学到知识并获得正反馈，不是为难玩家。**

- **正确选项应该"认真玩过就能选对"**：只要玩家在source处留意过信息，quiz中正确选项应该明显区别于错误选项
- **错误选项是"没注意会选的"，不是"故意误导"**：错误选项应该是笼统的、模糊的、想当然的回答，而非精心设计的陷阱
- **三个选项长度相近**：正确和错误选项的字数/细节量必须接近，不能靠"最长的就是对的"来猜答案。区别在于内容是否准确，不在于详略
- **错误选项也要有具体细节**：错误选项应包含看似合理但不准确的具体内容（错误的数字、张冠李戴的细节、现代常识的代入），而非空话套话
- **奖励才是重点**：选对后NPC多说的1-3句话才是真正的知识传递，quiz本身只是触发器
- **难度曲线平缓**：不追求"只有30%玩家能选对"的难度，目标是"70-80%认真玩家能选对"

## 六种合法触发模式

每个quiz节点**必须**属于至少一种：

| 代号 | 模式 | 一句话 | 适用角色 |
|------|------|--------|---------|
| A | 信任门槛 | "你够格听这些吗？" | 持有敏感信息的NPC |
| B | 情感共鸣 | "你真的懂我的痛吗？" | 情感驱动的NPC |
| C | 专业对等 | 专家按Zack水平调整沟通深度 | 专业身份NPC |
| D | 对话回声 | Zack在对话中自然引用前序信息 | 任何NPC |
| E | 防御反弹 | 嫌疑人试探Zack知道多少 | 有防御心理的NPC |
| F | 环境共振 | 场景环境元素触发观察选项 | 搜证场景 |

---

## 五种禁止模式

| 编号 | 禁止模式 | 说明 |
|------|---------|------|
| ❌1 | 老师提问 | NPC无理由突然变成考试出题者 |
| ❌2 | 突兀插入 | 正在说A话题突然切到B话题考查 |
| ❌3 | 无动机测试 | NPC本人都不关心答案 |
| ❌4 | 单句弹窗 | 三选一弹出后NPC只说一句就结束 |
| ❌5 | 重复测试 | 同一条信息在不同NPC处反复考 |

## State文件写入格式

### Source节点

写在对应NPC的talk条目下，或场景的evidence/搜证区块旁：

```yaml
connoisseur:
  - type: source
    content: "具体的信息内容描述"
    note: "L{X} {npc}_{talk} quiz#{编号} 考这个（来源说明）"
```

### Quiz节点

写在对应NPC的talk条目下：

```yaml
connoisseur:
  - type: quiz
    node: "#{编号}"
    trigger_mode: "D"  # 六种模式之一：A/B/C/D/E/F
    question: "对话情境描述（不是考题原文，是NPC说了什么导致Zack需要回应）"
    answer: "正确选项的内容（来源：L{X} {scene} {证据/对话出处}）"
    reward: "选对后NPC多说的1-3句奖励内容"
```

### 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| type | ✅ | `source` 或 `quiz` |
| content | source必填 | source节点的信息内容 |
| note | source必填 | 标注哪个quiz会考这个信息 |
| node | quiz必填 | 节点编号，格式 `#{编号}` |
| trigger_mode | quiz必填 | 六种合法触发模式之一（A-F） |
| question | quiz必填 | 对话情境描述 |
| answer | quiz必填 | 正确答案 + 信息来源标注 |
| reward | quiz建议填 | 选对后NPC的奖励台词（1-3句时代知识） |

### 放置位置

- Source和Quiz都写在**NPC的talk条目内**的 `connoisseur:` 字段下
- 如果是环境共振（模式F），写在**场景的evidence区块旁**
- 如果某个NPC无鉴赏力节点，写 `connoisseur: []`

## 对话草稿写入格式

在MD对话草稿中，鉴赏力节点使用以下格式：

```markdown
<!-- 鉴赏力 #{编号} | {领域徽章} {领域名} | 触发模式: {A-F} | 信息来源: {具体出处} -->

### {对话ID} `branches`
**扎克·布伦南** [{动作/表情描述}]
> {上下文台词——引出分支的自然对话}
> - ❶ {选项1：正确答案} → `{正确路径ID}`
> - ❷ {选项2：错误答案} → `{错误路径ID}`
> - ❸ {选项3：错误答案/中性} → `{错误路径ID}`

### {正确路径ID}
**{NPC名}** [{积极反应的动作/表情}]
> {选对反应：NPC态度积极}
> {奖励内容：1-3句时代知识，以NPC口吻说出}
→ 汇合至 `{汇合节点ID}`

### {错误路径ID}
**{NPC名}** [{冷淡反应的动作/表情}]
> {选错反应：NPC冷淡/敷衍，1句带过}
→ 汇合至 `{汇合节点ID}`
```

### MD格式要点

1. 元数据注释必须包含：编号、领域、触发模式、信息来源
2. 入口必须是标准 `branches` 分支
3. 分支选项是**Zack的台词**（Zack主动说），不是NPC的提问
4. 正确选项的措辞应是对source信息的**理解转述**，不是原文复读
5. 选对/选错路径必须汇合回主对话流
6. 奖励内容以**NPC的口吻和立场**说出，不是百科旁白

## 选对/选错反应风格

### 选对

| NPC类型 | 反应风格 |
|---------|---------|
| 防备型（Bernard、Morrison） | 微妙态度软化，"看来你不是随便来问问的" |
| 情感型（Margaret、Liam） | 情感打开，愿意说出更私密的内容 |
| 专业型（Foster） | 切换专业对等模式，用术语和细节交流 |
| 敌对型（Helen指证时） | 沉默或警惕，"你比我以为的知道得多" |

### 选错

| NPC类型 | 反应风格 |
|---------|---------|
| 防备型 | 关上话题，"这不是几句话能说清的" |
| 情感型 | 失望但不愤怒，"……算了" |
| 专业型 | 简化解释，跳过细节 |
| 敌对型 | 轻蔑，"看来你什么都不知道" |

## Source节点设计规则

1. **信息由NPC主动说出或环境自然呈现**——不能为埋source让NPC说出不符角色的话
2. **信息密度适中**——嵌入自然对话中，不突出也不隐藏。认真听能记住，走马观花会忽略
3. **Source和Quiz措辞有变化**——玩家需要"理解"而非"复读"

## 审查清单

设计完成后，对每个节点逐条检查：

### 系统层

- [ ] 信息来源在节点触发前玩家已接触过？
- [ ] 引用的证据ID在当前Loop范围内？
- [ ] source和quiz之间有足够距离（至少跨场景）？
- [ ] 以对话分支格式书写，不是考题？
- [ ] 选对/选错有明显内容差异？

### 自然度

- [ ] **主语**：是Zack主动展示，不是NPC提问？
- [ ] **动机**：NPC有角色层面的理由参与？
- [ ] **流程**：从当前话题自然生长，不是硬切？
- [ ] **模式**：属于六种合法模式之一（A-F）？
- [ ] **语气**：去掉鉴赏力标记后对话仍通顺？
- [ ] **NPC反应**：奖励台词是NPC口吻，不是百科旁白？

### 写入格式

- [ ] State文件中 `connoisseur:` 字段格式正确（type/content/note 或 type/node/trigger_mode/question/answer）？
- [ ] quiz节点标注了 `trigger_mode`（A-F之一）？
- [ ] MD草稿中元数据注释完整（编号、领域、触发模式、来源）？
- [ ] 正确选项是Zack的台词（主动展示），不是回答NPC的提问？
- [ ] 所有路径汇合回主对话流？
