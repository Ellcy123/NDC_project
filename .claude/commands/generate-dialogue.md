# State → 对话草稿生成工作流

根据 state 文件 + 知识池 + 角色人设，为指定循环生成完整的对话草稿 MD 文件。

---

## 使用方式

用户调用时指定循环编号，例如：`/generate-dialogue 1`（生成 Loop 1 的对话草稿）。

如果用户未指定循环编号，询问要生成哪个循环（1-6）。

---

## 输入文件

执行前必须读取以下文件：

### 1. State 文件（核心驱动）
`AVG/对话配置工作及草稿/前置配置/loop{N}_state.yaml`

包含：player_context、opening、scenes（含 NPC 条目）、expose、doubts。这是生成对话的蓝图。

### 2. 知识池
`AVG/对话配置工作及草稿/前置配置/npc_knowledge_pools.yaml`

NPC 的固定知识和盲区。State 文件中 `active_topics` 指定本循环要展示哪些主题。

### 3. 角色人设（NPC）
`AVG/对话配置工作及草稿/前置配置/characters/{npc_name}.md`

每个 NPC 的说话方式、核心性格、绝对不会做的事。**对话语气必须严格匹配人设。**

### 4. 主角人设
`剧情设计/00_世界观与角色/主角设计/`

- `Zack_Brennan.md` — Zack 的完整背景、性格特征、创伤来源、行为模式
- `Emma_OMalley.md` — Emma 的完整背景、与 Zack 的关系定位
- `Mickey_Donnelly.md` — Mickey 的背景（如涉及）

**Zack 和 Emma 是每个循环的核心角色，生成对话前必须读取他们的人设文档。** Zack 的对话应体现：
- 表面克制、内心激烈的双重性格
- 自嘲式幽默和刻薄毒舌
- 对正义的执着但不说教
- 小动作习惯（嚼糖、摸口袋里的铁盒子等）

### 5. 设计文档（补充参考）

根据章节查阅对应设计文档，确认剧情细节：
- EPI02: `剧情设计/Unit2/`
- EPI03: `剧情设计/Unit3/`

**绝不凭记忆编写剧情细节。**

### 6. 禁止参考已有对话草稿

**绝对不要**读取 `AVG/对话配置工作及草稿/Loop{N}_对话草稿.md` 或任何已有的对话成品。生成必须完全基于 state 文件 + 知识池 + 角色人设 + 设计文档，不受已有对话影响。

---

## 输出

输出到：`AVG/对话配置工作及草稿/生成草稿/Loop{N}_生成草稿.md`

如果文件夹 `生成草稿/` 不存在则创建。

## 生成规则

### 一、总体结构（按 state 文件段落顺序）

```
# Loop {N} - {循环主题}

## Talk: {opening talk}.json        ← state.opening
## Talk: {npc}_00X.json             ← state.scenes[].npcs（每个 NPC 一个 Talk 段落）
## Talk: {npc}_00X_repeat.json      ← 每个 NPC 配套的 repeat 对话
## Expose: loop{N}_{target}.json    ← state.expose
```

### 二、开篇对话（Opening Talk）

1. 读取 `opening` 段落的 purpose、characters、emma 的 knows/does_not_know
2. 开篇对话功能：
   - 承接上一循环的结果（L1 则是故事开端）
   - 交代本循环的调查方向（player_context.new_directions）
   - 自然引出自由探索阶段的目标
3. Emma 的语气参考 `characters/emma.md`
4. 开篇长度：10-20 句

### 三、NPC Talk 对话

对每个 NPC，按以下步骤生成：

**步骤 1：信息收集**
- 从 `npc_knowledge_pools.yaml` 拉取该 NPC 的 knowledge_pool
- 用 `active_topics` 过滤出本循环要展示的知识子集
- 读取 `awareness`（如有）了解实时认知变化
- 读取 `characters/{npc}.md` 确定说话风格
- **逐条列出该 NPC 的 `blind_spots`，作为写对话时的禁区清单**

**步骤 1.5：NPC 禁区清单（必做，写对话前完成）**

为每个 NPC 建立一份禁区清单，在写每句台词时对照检查：

1. 打开 `npc_knowledge_pools.yaml`，找到该 NPC 的 `blind_spots` 列表
2. 打开 state 文件，找到该 NPC 的 `withheld_topics` 列表
3. 将两者合并，形成**该 NPC 在本次对话中绝对不能提及的信息**
4. 在生成过程中，每写一句 NPC 台词都回查这份清单

**禁区不仅包含直接陈述，也包含间接暗示：**
- blind_spot: "Rose藏在Margaret鞋坊地下室"
  - ✗ NPC 不能说"地下室有人住" "修地下室给她住" 等任何暗示 Rose 与地下室关联的话
  - ✓ NPC 可以说"不知道她住哪儿" "没见她来过"
- blind_spot: "Frank的贷款具体条款（月息12%、原始金额600）"
  - ✗ NPC 不能说出任何具体数字
  - ✓ NPC 可以说"利息很高" "欠的钱越来越多" 等模糊感受

**如果 NPC 的台词中出现了禁区清单中的信息（无论直接还是间接），立即删除或改写。**

**步骤 2：对话结构设计**
- `player_inquiry` 分数决定对话重点：
  - 分数 8-10 的话题 → 主线内容，应占对话大部分篇幅
  - 分数 5-7 的话题 → 可作为分支选项
  - 分数 2-4 的话题 → 可省略或一句带过
- 分支设计：2-3 个分支，指向不同信息维度（身份/时间线/事件/情绪/背景）

**步骤 2.5：分支选项的视角校验（必做）**

分支选项代表 **Zack 此刻会说的话**，必须通过以下校验：

1. **信息可达性检查** — 选项中涉及的每一个具体信息（人名、事件、细节），Zack 是否已经知道？
   - 检查来源：`player_context.known_facts` + 本循环 opening 中获得的信息 + 本场景之前的对话已获取的信息
   - 如果 Zack 不可能知道某个细节 → **禁止**写进选项文本
   - 例：Zack 不知道 Vinnie 对 Sarah 做过什么 → ✗ "再说说 Vinnie 对 Sarah 做了什么"
   - 例：Zack 已知有人上门催收 → ✓ "催收的人对你们做了什么？"

2. **具体度控制** — 选项应该是**泛化的好奇/追问**，不是精确的信息复述
   - ✗ "Leonard 的三年空白期是在哪里度过的？"（Zack 还不知道有空白期）
   - ✓ "Leonard 这个人你了解多少？"（泛化提问，NPC 回答时才揭示细节）
   - ✗ "Frank 的举报材料里有多少受害者？"（Zack 还没看到材料）
   - ✓ "Frank 最近有没有什么反常举动？"（基于已知的"Frank 在调查"来提问）

3. **好奇心驱动** — 选项应体现 Zack 基于**已有线索**产生的合理好奇
   - 已知"有人来要过钱" → "他们来要钱的时候，有没有动手？"
   - 已知"Frank 欠了银行钱" → "Frank 到底欠了多少？"
   - 已知"Leonard 帮 Frank 填表" → "Leonard 平时怎么跟客户打交道？"

**每设计一个分支选项，都必须自问：Zack 凭什么知道这件事？凭什么会问这个问题？如果答不上来，就改成更泛化的表述。**

**步骤 3：证词与证据嵌入**
- `testimony_ids` 中的每个 ID → 在对话中对应位置插入 `get` 标记
- 两次 `get` 之间至少隔 3-5 句普通对话
- 多个证据可分散到不同分支路径中
- `⚠谎言:` 标注的证词 → NPC 在说这句话时的语气要自然，不能让玩家一听就知道是谎言

**步骤 4：鉴赏力节点**

**核心原则：答案永远来自游戏内信息积累，不考验玩家的现实知识。鉴赏力 quiz 永远是 NPC 考验 Zack（玩家），不是 Zack 考 NPC。**

- `type: source` → NPC 在对话中自然提及该信息，加 HTML 注释 `<!-- 后面要考：{信息内容} -->`。这是为后续 quiz 埋下信息种子，玩家需要记住。
- `type: quiz` → 生成鉴赏力分支，规则如下：
  1. **由 NPC 发起考验**：NPC 在对话中自然地抛出问题、质疑、或制造需要判断的情境（如"你连这个都不知道？"、"那你说说看……"），Zack 做出回应（分支选项）
  2. **HTML 注释**：分支前加 `<!-- 鉴赏力 #XXX | 领域 | 信息来源: 具体出处 -->`
  3. **分支格式**：固定 **3 个选项**（1 正确 + 2 迷惑），Zack 的回答作为分支选项
     - ❶ 正确选项（答案与 state 中 `answer` 字段一致）
     - ❷ 迷惑选项 A（看似合理但细节错误）
     - ❸ 迷惑选项 B（另一个方向的错误）
  4. **迷惑选项设计规则**：
     - ✗ 禁止"还在查""我不确定""不知道"等逃避式选项——这不是考验，是送分
     - ✗ 禁止明显荒谬的选项——玩家一眼就能排除的不叫迷惑
     - ✓ 迷惑选项必须**看起来有道理**：用真实存在的游戏内信息，但**张冠李戴**或**混淆细节**
     - 示例：quiz 问"催收的人长什么样？"
       - ❶ "一个金发年轻人带着一个意大利壮汉"（正确——来自 L2 O'Hara 描述）
       - ❷ "一个戴眼镜的中年人，独自来的"（迷惑——混入 Leonard 的外貌特征，但催收的不是他本人）
       - ❸ "两个意大利人，都很壮"（迷惑——把 Vinnie 的特征复制到了同伴身上）
  5. **正确路径**（2-3 句）：NPC 认可 + 展开信息，然后 **Zack 内心解读**（1 句独白，将正确答案转化为推理线索，衔接后续对话）
  6. **错误路径**（1 句）：NPC 冷淡、失望或轻蔑地带过（根据 NPC 性格），但不直接说出正确答案
  7. **三条路径汇合**到同一节点继续对话
  8. **信息距离**：quiz 考的内容必须是玩家在之前 Loop 或当前 Loop 其他场景中已经接触过的信息，禁止"同一段对话中刚说完就考"
  9. **证据 ID 范围**：Loop N 的鉴赏力节点只能引用 21xx ~ 2Nxx 范围内的证据

**步骤 5：对话长度**
- 标准 NPC Talk：15-30 句（含分支）
- 核心 NPC（player_inquiry 分数高）：可到 40 句
- 过渡型 NPC（仅 1 个信息点）：10-15 句

### 四、Repeat 对话

每个 NPC 的 Talk 之后，紧跟一个 `_repeat.json` 段落：
- 简短的寒暄开头（2-3 句）
- 分支选项直接指回首次对话中的分支路径 ID
- 末尾有退出选项

### 五、Expose 对话

按 state.expose 的 round_1 ~ round_N 逐轮生成：

1. **开场**（3-5 句）：Zack 进入场景，说明来意，嫌疑人初始反应
2. **每轮结构**：
   - 嫌疑人重申/说出谎言（`Lie` 标记）
   - 标注可用证据（🎯）
   - Zack 出示证据反驳（2-4 句交锋）
   - 嫌疑人抵抗 → 被击破 → 承认部分事实
   - 嫌疑人主动缩小范围/淡化说辞（新谎言，止损）
   - Zack 确认/记录嫌疑人的说法
   - 嫌疑人加码承诺（进入下一轮 Lie 步骤）
3. **结局**：最终揭示 + 获取证据（如有）+ 嫌疑人的条件/威胁
4. **谎言由嫌疑人主动说出**（止损），不是 Zack 喂话后被动否认
5. **实物引用准确性**：Expose 引用物证时，核对 `ItemStaticData.Describe` 和 `ArtRequirement`，对话描述必须与美术描述一致

### 六、ID 编码

- 对话 ID 格式：9 位数字 `{loop}{scene}{sequence}`
  - opening 的 scene 用 state.opening.scene_id 的后两位
  - 自由探索 NPC 的 scene 用 scenes[].id 的后两位
  - expose 的 scene 固定为 `100`（即 `{loop}100{sequence}`）
- 分支 ID：主 ID 的千位改为分支编号（❶=1xx, ❷=2xx, ❸=3xx）
- 汇合点 ID：分支结束后的下一个百位整数

## MD 格式规范

严格遵循以下格式：

```markdown
# Loop {N} - {主题}

## Talk: {filename}.json
> 场景：{cnLocation}

### {9位ID}
**{说话人中文名}** [{情绪/表情}]  ← 方括号内只写情绪/表情，不写肢体动作
> 对白内容

### {ID} `get` → {证据ID}
**{说话人}**
> 对白内容
> 📋 获取证据: {ID}「{证据描述}」

### {ID} `branches`
**{说话人}** [{情绪}]
> {内心独白或过渡}
> - ❶ 选项文本 → `{跳转ID}`
> - ❷ 选项文本 → `{跳转ID}`

### {ID} `Lie` → 成功跳转 `{ID}` (Round N)
**{说话人}**
> 谎言内容
> 🎯 可用证据: {ID}「{名称}」, {ID}「{名称}」

### {ID} `end`
**{说话人}**
> 结束语
```

分支用 `---` 分隔，末尾标注 `→ 汇合至 {ID}`。
HTML 注释用于鉴赏力标注：`<!-- 鉴赏力 #XXX | 领域 | 信息来源: 具体出处 -->`

## 审查清单（生成后自检）

1. **知识边界（逐条对照）** — 对每个 NPC 的每句台词，逐条检查 `blind_spots` 和 `withheld_topics`：NPC 是否说了/暗示了禁区中的任何信息？包括间接暗示（如提到"地下室"间接暴露 Rose 的藏身处）。NPC 只能说 active_topics 范围内的知识 + awareness；不说 withheld_topics 和 blind_spots 中的内容；不剧透本轮 Expose 结论和后续 Loop 发现
2. **信息分配** — 同 Loop 内不同 NPC 不传递重复信息；每条信息有唯一"信息所有者"
3. **证据获取覆盖** — state 中 testimony_ids 的每个 ID 都有对应 `get` 句
4. **实物引用准确性** — 提到物品时核对 `ItemStaticData.Describe` 和 `ArtRequirement`，对话描述必须与美术描述一致
5. **角色语气** — 对照 characters/*.md 确认无"串味"；方括号内只写情绪/表情，不写肢体动作
6. **对话链完整性** — 无断链；分支 ID 有效；末尾有 `end`；分支必须汇合
7. **获取节奏** — 两次 `get` 之间至少隔 3-5 句
8. **ID 编码** — 9 位数字，格式正确，无重复
9. **Expose 谎言驱动** — 谎言由嫌疑人主动说出（止损）；不是被喂话；证据 ID 真实存在；描述与实际一致
10. **鉴赏力节点** — source 自然嵌入 + HTML 注释（含编号/领域/来源）；quiz 由 NPC 考 Zack（不是反过来）；正确路径含 Zack 内心解读；信息距离足够；正确/错误路径都有且汇合
