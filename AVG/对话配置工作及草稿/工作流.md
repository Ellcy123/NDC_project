# 对话配置工作流

## 总览

```
MD草稿（中文）→ 审阅修改 → 同步脚本 → JSON → 英文翻译（后续批量）
```

- 每个 Loop 一个 MD 文件（包含该 Loop 所有 Talk + Expose 对话）
- MD 只写中文，英文后续批量翻译
- 修改完 MD 后用同步脚本回写到 JSON

### 修改顺序：先 MD 后 JSON

**修改对话时必须先改 MD 草稿，再同步到 JSON。** MD 是人类可读的主文档，上下文连贯、便于审阅对话流。JSON 是程序消费的派生文件。
1. 在 MD 草稿中修改中文对白（利用 MD 的上下文可读性确认逻辑通顺）
2. 确认 MD 无误后，将改动同步到对应的 JSON 文件（补充英文翻译、speaker 字段等）

## MD 格式规范

### 基本结构

```markdown
# Loop X - [循环主题]

## Talk: emma_00X.json
> 场景：[cnLocation]

### [id]
**[说话人]** [情绪/表情]
> 对白内容
```

### 字段说明

| MD 元素 | 对应 JSON 字段 | 说明 |
|---------|---------------|------|
| `### [id]` | `id` | 对话句 ID，四级标题 |
| `**说话人**` | `cnSpeaker` | 加粗显示 |
| `[情绪]` | `cnAction` | 方括号内只写情绪/表情，不写肢体动作 |
| `> 对白` | `cnWords` | blockquote 格式 |
| `> 场景：` | `cnLocation` | 文件开头标注一次 |

### 特殊脚本标记

| 标记 | 对应 JSON `script` | 说明 |
|------|-------------------|------|
| `` `end` `` | `"end"` | 对话结束 |
| `` `get` `` | `"get"` | 获取证据/道具 |
| `` `branches` `` | `"branches"` | 分支选择 |
| `` `change_scene` `` | `"change_scene"` | 切换场景 |
| `` `Lie` `` | `"Lie"` | 指证谎言点（Expose 专用） |

## 分支展示格式

### 分支入口

分支句在 ID 后标注 `` `branches` ``，对白下方用列表展示选项：

```markdown
### 209001504 `branches`
**扎克·布伦南** [眉头紧锁]
> （记下编号，眉头越拧越紧。）
> - ❶ 那个意大利人的外貌 → `209001100`
> - ❷ Frank最近的身体状况 → `209001300`
```

### 分支内容

每个分支用水平线分隔，开头注释标明所属分支：

```markdown
---
<!-- 分支❶：意大利人外貌 -->

### 209001100
**扎克·布伦南**
> 那个上门催收的意大利人，您还记得他长什么样吗？

### 209001101
**欧哈拉太太** [回忆]
> 很高，很壮……

...（分支对话）...

### 209001109
**扎克·布伦南**
> 我记下了。
→ 汇合至 `209001400`
```

### 分支汇合

分支末尾标注 `→ 汇合至 [id]`，汇合点后恢复正常顺序。

## 对话创作规则

### 知识边界

- 角色只能引用**已有证据支撑的信息**（TestimonyItem、ItemStaticData 中的物品、前序 Loop 已确认的事实）
- 不能凭空添加未经证据佐证的细节（如地理来源、家庭背景等）
- 审查时逐句核对：Zack 提出的每个主张，都必须能在证据表中找到对应条目
- **Expose 引用实物证据时**，必须查阅 `ItemStaticData` 中该物品的 `Describe` 和 `ArtRequirement` 字段，对话中描述的细节（文字、外观、标记等）必须与美术描述完全一致，不可凭想象添加物品上不存在的内容

### Expose 谎言驱动原则

- **谎言由嫌疑人主动说出**，不是 Zack 引导/喂话后被动否认
- 第 2、3 轮的谎言应来自嫌疑人**试图止损**——为了解释上一轮被揭穿的事实，嫌疑人主动缩小/淡化说辞，从而说出新的谎言
- Zack 的角色是**提出证据打破谎言**，而非提出指控让嫌疑人否认

**反例（错误）：**
```
Zack: "你和Vinnie之间有资金往来。"     ← Zack喂话
Leonard: "荒谬，没有。"                ← 被动否认 = 弱
```

**正例（正确）：**
```
Leonard: "我承认我以前叫Russo...但我们之间没有金钱关系。"  ← 主动撇清（止损）
Zack: "你说没有金钱关系。"                                ← 只是复述确认
Leonard: "没有，一分钱都没有。"                            ← 自己加码承诺 = Lie
→ Zack 用连号钞票证据打破
```

### Expose 结构模板

```
Round N-1 揭穿 → 嫌疑人被迫承认部分事实
                → 嫌疑人主动缩小范围 / 淡化说辞（新谎言）
                → Zack 确认/记录嫌疑人的说法
                → 嫌疑人加码承诺（Lie 步骤）
Round N    → Zack 出示证据打破新谎言
```

## 审查

每个 Loop 的 Talk + Expose 写完后，按以下 7 项逐一审查。

### 1. 知识边界

- 每个 NPC 只能说出**自己知道的事**（亲身经历、亲眼所见）
- Zack 只能引用**已获得的证据**（当前 Loop 及之前 Loop 的 TestimonyItem + 物品）
- 不能剧透**本轮 Expose 的结论**（那些是指证时才揭露的事实）
- 不能剧透**后续 Loop 的关键发现**（参照知识边界约束表）

### 2. 信息分配

- **同一 Loop 内不同 NPC 的对话不应传递重复信息**
- 每条信息应有唯一的"信息所有者"——由最合适的 NPC 首次说出
- 如果两个 NPC 都知道同一件事，各自的描述应从**不同角度/细节**切入，而非简单重复
- 审查方法：列出每个 Talk 文件传递的核心信息点，检查是否有交叉重复

| 信息点 | 应由谁说 | 不该重复出现在 |
|--------|---------|--------------|
| 例：Vinnie 用连号新钞票 | Tony（亲眼所见） | O'Hara 的对话中不该重复同样细节 |
| 例：Frank 被催收威胁 | O'Hara（亲身经历） | Moore 的对话中不该出现同样描述 |

### 3. 证据获取覆盖

- 对照设计文档，检查该 Loop 应获取的所有 TestimonyItem 是否都有对应的 `script: "get"` 句
- `get` 句的 `ParameterInt` 必须指向正确的 TestimonyItem ID
- 证据获取的上下文必须自然——NPC 说出该信息时，玩家自然获得对应证词

### 4. 实物引用准确性

- 对话中提到具体物品时，必须核对 `ItemStaticData` 的 `Describe` 和 `ArtRequirement` 字段
- 描述的细节（文字内容、外观、标记等）必须与美术描述完全一致
- 不可凭想象添加物品上不存在的内容（如编造笔记本上的文字、凭空添加物品特征）

### 5. 角色语气一致性

- 每个 NPC 的说话方式必须符合角色设定（语气、用词习惯、情绪表达）
- 审查要点：
  - 是否有"串味"（A 角色说了 B 角色风格的话）
  - 情绪变化是否有合理的上下文铺垫
  - Zack 的提问是否符合侦探的专业性和克制感

### 6. 对话链完整性

- `next` 字段指向的 ID 必须存在，无断链
- 分支 `branches` 的 `ParameterInt` 必须指向有效的 Talk ID
- 对话末尾必须有 `script: "end"` 或 `next: "-1"`
- 分支最终必须汇合到共同节点

### 7. Expose 谎言驱动 + 证据匹配

- 谎言必须由嫌疑人**主动说出**（止损），不是 Zack 喂话后被动否认
- 每轮之间的转场：嫌疑人被揭穿后**主动缩小范围**，引出下一轮谎言
- `Lie` 步骤的 `ParameterStr0` 中列出的证据 ID 必须真实存在
- Zack 引用证据时的描述必须与 `ItemStaticData` / `TestimonyItem` 的实际内容一致

### 8. 鉴赏力节点

完整设计文档见 `docs/游戏系统/核心玩法/鉴赏力系统.md`。以下是对话草稿阶段必须遵守的规则。

#### 核心原则：答案必须来自玩家已接触的游戏内信息

**答案永远来自游戏内信息积累，不考验玩家的现实知识。** 两种合法来源：

1. **对话** — 当前 Loop 或之前 Loop 中 NPC 已经说过的内容
2. **证据/道具** — 玩家已收集的 TestimonyItem 或 ItemStaticData 中的描述

#### 证据 ID 编码（EPI02）

物品 ID 前两位标识所属循环，决定了该证据何时可用：

| ID 前缀 | 所属循环 | 最早可引用时间 |
|---------|---------|-------------|
| 21xx | Loop 1 | Loop 1 起 |
| 22xx | Loop 2 | Loop 2 起 |
| 23xx | Loop 3 | Loop 3 起 |
| 24xx | Loop 4 | Loop 4 起 |
| 25xx | Loop 5 | Loop 5 起 |
| 26xx | Loop 6 | Loop 6 起 |

写 Loop N 的鉴赏力节点时，只能引用 **21xx ~ 2Nxx** 范围内的证据，以及 Loop 1 ~ Loop N 中已发生的对话。

#### 信息距离要求

- 信息来源与测试点之间应有**足够的距离**——最好跨场景或跨 Loop
- **禁止**在同一段对话中"刚说完就考"（NPC 刚说了一个细节，下一句就测试这个细节 = 没有难度，形同虚设）
- 理想情况：Loop N 的鉴赏力节点测试 Loop 1 ~ Loop N-1 的旧信息，或者当前 Loop 中**其他场景**已给出的信息

#### 对话格式（非考题格式）

鉴赏力节点必须写成**标准对话分支**，不是考题。玩家不应感觉到"这是一道试题"。

**错误写法（考题格式）：**
```
[鉴赏力节点 #001]
问题：Frank拿什么做担保？
A. 退休金
B. 房产 ← 正确
C. 鞋坊经营权
```

**正确写法（对话分支格式）：**
```markdown
<!-- 鉴赏力 #001 | 🕵️ Street Smart | 信息来源: L2 ohara_001 209001006 -->

### 206001050 `branches`
**扎克·布伦南** [盯着贷款协议]
> 一个退休码头工人，您觉得他能拿什么做抵押？
> - ❶ 他的退休金 → `206001051`
> - ❷ 他名下的房产 → `206001052`
> - ❸ 鞋坊的经营权 → `206001051`

### 206001052
**哈罗德·摩尔** [手指一顿——判断比他预期的精准]
> ……看来您对南区居民的情况很了解。是的，房产是唯一有价值的抵押物。
→ 汇合至 `206001008`

### 206001051
**哈罗德·摩尔** [微笑摇头]
> 具体的担保条款属于银行内部信息。
→ 汇合至 `206001008`
```

#### 格式要点

| 元素 | 要求 |
|------|------|
| 元数据注释 | `<!-- 鉴赏力 #编号 \| 领域 \| 信息来源: 具体出处 -->` |
| 入口 | 标准 `branches` 分支，由 NPC 自然触发或 Zack 自然提出 |
| 选对反应 | NPC 态度积极，**多说 1-3 句**世界观/细节内容（奖励性信息） |
| 选错反应 | NPC 冷淡/敷衍，**1 句带过**，不给额外信息 |
| 汇合 | 必须汇合回主对话流，不影响主线推进 |

#### 审查清单

- [ ] 每个节点的信息来源是否在节点触发前**玩家已经接触过**？
- [ ] 引用的证据 ID 是否在当前 Loop 或之前 Loop 的范围内（21xx ~ 2Nxx）？
- [ ] 信息来源与测试点之间是否有足够距离（不是同一段对话刚说完就考）？
- [ ] 是否以对话分支格式书写（不是考题/问答格式）？
- [ ] 选对/选错的 NPC 反应是否有明显的内容差异（不只是表情不同）？
- [ ] 节点是否自然融入对话流（玩家不会感觉"这是一道考试题"）？
